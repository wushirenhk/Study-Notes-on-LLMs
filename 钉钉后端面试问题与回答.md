**Q: 秒杀场景下删除缓存引发的问题？**

| 问题 | 触发场景 | 解决方案 |
|------|---------|---------|
| **缓存击穿** | 热点key过期瞬间，大量请求同时打到DB | 互斥锁 / SETNX / 永不过期+异步更新 |
| **缓存雪崩** | 大量key同时过期 / Redis集群崩溃 | 过期时间+随机偏移 / 多级缓存 / 熔断降级 |
| **缓存穿透** | 查询不存在的数据（恶意刷接口） | 空值缓存+短TTL / 布隆过滤器 / 用户行为限流 |

**解决方案详解**：

```python
# 1. 互斥锁防止缓存击穿
async def get_stock_with_lock(product_id):
    lock_key = f"lock:stock:{product_id}"
    lock_val = str(uuid4())
    
    # SETNX获取锁
    acquired = await redis.set(lock_key, lock_val, nx=True, ex=10)
    if acquired:
        try:
            # 只有获取到锁的请求查DB
            db_val = await db.query(...)
            await redis.set(f"stock:{product_id}", db_val, ex=300)
            return db_val
        finally:
            # 释放锁（Lua脚本保证原子性）
            await redis.eval("if redis.call('get',KEYS[1])==ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end", [lock_key], [lock_val])
    else:
        # 没拿到锁，等待后重试
        await asyncio.sleep(0.1)
        return await get_stock_with_lock(product_id)

# 2. 永不过期+异步更新解决击穿
async def get_stock_never_expire(product_id):
    val = await redis.get(f"stock:{product_id}")
    if val:
        return int(val)
    
    # 缓存miss，读取DB
    db_val = await db.query(...)
    # 写入时设置永不过期
    await redis.set(f"stock:{product_id}", db_val)
    
    # 触发异步更新（如果TTL过期标记存在）
    asyncio.create_task(async_refresh_stock(product_id))
    return db_val
```

```python
async def扣减库存(product_id, quantity, request_id):

    idempotent_key = f"idempotent:{request_id}"
    
    # 2. 检查是否已处理
    exists = await redis.exists(idempotent_key)
    if exists:
        return await redis.get(idempotent_key)  # 返回之前的结果
```

**Q: RAG检索质量低的原因与解决方案？**

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 漏召 | Query改写不充分、Embedding不匹配领域 | Query Rewrite、领域微调Embedding |
| 误召 | 分块过大噪声多、Top-K过大 | 语义分块、重排序过滤 |
| 排序差 | 单一向量检索精度不足 | 混合检索(Hybrid) + Cross-Encoder Rerank |
| 上下文不足 | 召回内容太少无法支撑回答 | Parent-Document检索、多路召回 |

```python
# 多路召回 + RRF融合
async def hybrid_retrieve(query: str, top_k: int = 20):
    # 向量检索
    vector_results = await vector_store.search(query, k=top_k*2)
    
    # BM25关键词检索
    bm25_results = await bm25_index.search(query, k=top_k*2)
    
    # RRF融合 (Reciprocal Rank Fusion)
    fused_scores = defaultdict(float)
    for rank, doc in enumerate(vector_results):
        fused_scores[doc.id] += 0.6 / (60 + rank)
    for rank, doc in enumerate(bm25_results):
        fused_scores[doc.id] += 0.4 / (60 + rank)
    
    # 按融合分数排序
    sorted_ids = sorted(fused_scores, key=fused_scores.get, reverse=True)[:top_k]
    return [get_doc_by_id(id) for id in sorted_ids]
```

**Q: Agent中向量检索的原理与结构？**

```python
# Agent中的向量检索用于Tool/Memory召回
class AgentVectorRetrieval:
    """Agent系统中向量检索的核心应用"""
    
    # 1. Tool路由：找到最相关的工具
    async def retrieve_tools(self, query: str, tools: List[Tool]) -> List[Tool]:
        # Query向量化
        query_emb = await self.embedder.embed(query)
        
        # 并行计算所有Tool描述与Query的相似度
        similarities = []
        for tool in tools:
            tool_emb = await self.embedder.embed(tool.description)
            sim = cosine_sim(query_emb, tool_emb)
            similarities.append((tool, sim))
        
        # 排序返回Top-K
        return [t for t, _ in sorted(similarities, key=lambda x: x[1], reverse=True)[:5]]
    
    # 2. 记忆召回：从历史中召回相关内容
    async def retrieve_memory(self, query: str, memory_embeddings: List[Memory]) -> List[Memory]:
        query_emb = self.embedder.embed(query)
        scores = [cosine_sim(query_emb, m.embedding) for m in memory_embeddings]
        top_indices = np.argsort(scores)[-10:][::-1]
        return [memory_embeddings[i] for i in top_indices if scores[i] > 0.7]
```

**Q: MCP的定义及在Agent中的优缺点？**

| 方面 | MCP (Model Context Protocol) |
|------|------------------------------|
| **定义** | Anthropic提出的AI模型与外部工具/数据源交互的标准协议，类似AI世界的"USB接口" |
| **优点** | 统一协议一次开发处处运行、安全隔离（独立进程）、丰富生态（文件/Git/DB等预构建Server） |
| **缺点** | 学习成本、版本兼容性、调试复杂、上下文增长（每个MCP调用携带完整工具定义） |

**MCP导致上下文过长的原因**：

```python
# 问题：每次调用都把所有MCP Server的完整Schema传入Context
MCP_CONTEXT_BLOAT = """
每次请求都携带：
- MCP Server列表（可能10+个）
- 每个Server的所有Tool定义（每个Tool有name/description/parameters）
- 历史调用记录

例如：10个Server × 平均20个Tool × 每个Tool 200 tokens = 40,000 tokens 仅用于MCP！
"""

# 解决方案：分层加载
class MCPToolManager:
    def get_relevant_tools(self, task: str, top_k: int = 5) -> List[Tool]:
        # 只加载与当前任务最相关的K个工具
        task_emb = self.embedder.embed(task)
        tool_embs = [self.embedder.embed(t.description) for t in self.all_tools]
        scores = [cosine_sim(task_emb, te) for te in tool_embs]
        top_indices = np.argsort(scores)[-top_k:][::-1]
        return [self.all_tools[i] for i in top_indices]
```

```java
// JDK7: 数组 + 链表 (Entry)
// JDK8: 数组 + 链表 + 红黑树 (TreeNode)

/*
 * JDK8 优化点：
 * 1. 链表长度 >= 8 时转为红黑树（O(n) → O(log n)）
 * 2. 红黑树节点数 <= 6 时退回链表

 * 4. 扩容优化：JDK8不再重新hash每个元素，而是利用高位运算判断是否需要移动
 *    新位置 = 原位置 + 原容量（原容量是2的幂次，高位bit直接决定位置）
 */
```

```java
/*
 * 对于基本类型（int[], double[]等）：
 *   → DualPivotQuicksort（双轴快排）
```

### 3.3 缓存经典问题

**Q: 缓存击穿的触发场景与通用解决方法？**

| 场景 | 触发条件 | 解决方案 |
|------|---------|---------|
| **缓存击穿** | 热点key过期瞬间大量并发请求打到DB | 互斥锁 / SETNX / 永不过期+异步更新 |
| **缓存雪崩** | 大量key同时过期 / Redis宕机 | 过期时间+随机偏移 / 多级缓存 / 熔断降级 |
| **缓存穿透** | 查询不存在的数据（恶意/无意） | 空值缓存+短TTL / 布隆过滤器 / 限流 |

```python
# 布隆过滤器解决缓存穿透
class BloomFilter:
    """用布隆过滤器判断key是否可能存在"""
    
    def __init__(self, size: int, hash_count: int):
        self.size = size
        self.hash_count = hash_count
        self.bit_array = [False] * size
    
    def add(self, key: str):
        for seed in range(self.hash_count):
            idx = hash(key + str(seed)) % self.size
            self.bit_array[idx] = True
    
    def might_contain(self, key: str) -> bool:
        for seed in range(self.hash_count):
            idx = hash(key + str(seed)) % self.size
            if not self.bit_array[idx]:
                return False  # 一定不存在
        return True  # 可能存在

# 使用示例
bf = BloomFilter(10000000, 7)

async def get_data(key: str):
    if not bf.might_contain(key):
        return None  # 布隆过滤器说一定没有，直接返回
    
    # 可能存在，查缓存
    val = await redis.get(key)
    if val:
        return val
    
    # 查DB
    val = await db.get(key)
    if val:
        await redis.setex(key, 300, val)
        bf.add(key)
    else:
        # 空值也要加入过滤器（防止穿透）
        await redis.setex(key, 60, "")
        bf.add(key)
    
    return val
```

```python
async def consume_with_ack(channel, callback):
    # 1. 设置QoS（预取数量）
    await channel.basic_qos(prefetch_count=10)
    
    # 2. 手动ACK
    async def on_message(body, delivery_tag):
        try:
            result = await process_message(body)
            if result.success:
                await channel.basic_ack(delivery_tag)

                await channel.basic_nack(delivery_tag, requeue=True)
```

```python
核心思想：
1. 先用K-Means将所有向量聚类为N个簇（如1024个）
2. 每个向量归属于某个簇，建立 簇ID → [向量列表] 的倒排索引

                    查询向量
                        ↓
             ┌──────────────────────┐
             │   K-Means 聚类中心     │
             │  [c0] [c1] [c2] ... [c1023]
             └──────────────────────┘
                        ↓
             找到最近的 top N 个簇

                        ↓
             ┌──────────────────────┐
             │  倒排索引 (Inverted File) │
             │  c5 → [v1, v15, v23...]
             │  c12 → [v3, v8, v19...]
             │  c87 → [v2, v45, v66...]
             └──────────────────────┘
                        ↓
             在N个簇中搜索最近邻
             (用PQ压缩向量，快速距离计算)

- Flat (暴力搜索): O(N) = 1,000,000 次距离计算
```

```python
1. 重传机制
   - 超时重传：发送方等待ACK超时未收到，重发数据包
   - 快速重传：收到3个重复ACK，立即重发丢失的包（不等超时）

   - 累计确认：ACK N表示N之前所有数据都收到了

   - TCP头部有序列号，接收方按序列号重组

4. 流量控制
   - 滑动窗口：接收方告诉发送方"我还剩多少缓冲区"
   - 防止发送方速度过快导致缓冲区溢出

5. 拥塞控制
   - 慢启动：从小窗口开始，指数增长
   - 拥塞避免：达到阈值后线性增长
   - 拥塞发生：乘性减小，快速恢复
   - 算法：Reno / Cubic / BBR
```

### 6.4 大模型算法基础

**Q: Transformer、Self-Attention的原理？**

```python
# Self-Attention 计算过程
SELF_ATTENTION = """
输入：X (seq_len, d_model)  — 每个token的embedding

Step 1: QKV投影
  Q = X @ W_q  → (seq_len, d_k)
  K = X @ W_k  → (seq_len, d_k)
  V = X @ W_v  → (seq_len, d_v)

Step 2: 注意力分数
  scores = Q @ K.T / sqrt(d_k)  → (seq_len, seq_len)
  每一行i表示token_i对所有token的注意力权重

Step 3: Softmax归一化
  attention_weights = softmax(scores, dim=-1)

Step 4: 加权求和
  output = attention_weights @ V  → (seq_len, d_v)

Multi-Head：分h个头各自计算，最后拼接
  每个头关注不同的语义子空间（语法/实体/关系等）
"""
```

**Q: KV Cache的原理？Flash Attention的原理？**

```python
# KV Cache（键值缓存）
KV_CACHE = """
问题：自回归生成时，每个token都要重新计算对所有历史token的attention
      浪费：前面token的K/V在每一步都重新计算

KV Cache优化：
- 首次生成token_1：计算完整的Q, K, V
- 生成token_2：Q_2是新计算的，K_2/V_2是新计算的
              但K_1, V_1直接从缓存读取，不需要重新计算

        cache_K = [K_1, K_2, ..., K_t-1]  ← 只需append K_t
        cache_V = [V_1, V_2, ..., V_t-1]  ← 只需append V_t

效果：每步计算量从 O(n²d) 降为 O(nd)（n=序列长度，d=维度）
      生成速度提升约 3-5倍
"""

# Flash Attention
FLASH_ATTENTION = """
问题：标准Attention需要 O(n²) 显存存储注意力矩阵（n=序列长度）
      长序列（如65K）显存爆炸

核心思想：用tiling分块计算 + 在线算法，减少显存占用

原理：
1. 分块：把softmax分解为逐块计算
2. 在线算法：边读入数据边累积统计量（max, sum）
3. 分块结果与全局结果数学等价，但不需要完整矩阵

效果：
- 显存：O(n²) → O(n)（几乎线性）
- 计算量：不变（甚至略有增加，因重复计算）
- 可处理65536+长度的序列

当前已成为LLM推理的事实标准（vLLM/TGI都基于此优化）
"""
```
