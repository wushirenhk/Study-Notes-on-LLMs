## 一、基础概念：RAG 与 LangGraph

### 1. 介绍一下RAG？它的核心组件和工作流程是什么？

**A：**

**RAG = Retrieval Augmented Generation（检索增强生成）**

RAG是一种让大模型"先检索、再生成"的技术架构，解决模型"知识不足"或"知识过时"的问题。

**为什么需要RAG**：

| 问题       | 直接问LLM              | RAG                        |
| ---------- | ---------------------- | -------------------------- |
| 知识实时性 | 依赖训练数据，可能过时 | 实时检索最新文档           |
| 事实准确性 | 可能幻觉               | 基于检索结果生成，有据可查 |
| 知识覆盖   | 受限于模型参数量       | 可扩展至海量外部文档       |
| 可溯源性   | 黑盒，不知道答案来源   | 附带引用来源               |

**RAG核心组件**：

```
┌─────────────────────────────────────────────────────────┐
│                      RAG Pipeline                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Document Processing（文档处理）                     │
│     原始文档 → 解析 → 分块(Chunking) → 向量化           │
│     ├── MinerU / Unstructured / PDFPlumber              │
│     └── Chunk策略：固定大小 / 语义分割 / 递归字符       │
│                                                         │
│  2. Vector Store（向量数据库）                          │
│     Embedding → 索引 → 存储                             │
│     ├── Milvus / Pinecone / Qdrant / Chroma           │
│     └── 索引类型：HNSW / IVF-PQ / Flat                 │
│                                                         │
│  3. Retrieval（检索阶段）                               │
│     用户Query → 向量化 → 相似度搜索 → Top-K召回        │
│     ├── 向量检索：语义相似度                            │
│     ├── 关键词检索：BM25 / TF-IDF                       │
│     └── 混合检索：RRF / 学到的重排序                    │
│                                                         │
│  4. Generation（生成阶段）                              │
│     召回文档 + Query → 组装Prompt → LLM → 生成答案      │
│     └── 关键：如何组装上下文（compression/rerank）     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**RAG工作流程**：

```
用户提问: "张三的季度销售额是多少？"
     ↓
Query向量化: embed("张三的季度销售额是多少？")
     ↓
向量检索: 在向量库中找最相似的K个Chunk
     ↓
Top-K召回: [文档A片段, 文档B片段, ...]
     ↓
Prompt组装:
┌────────────────────────────────────────┐
│ 基于以下文档回答用户问题：              │
│ [召回的文档片段1]                       │
│ [召回的文档片段2]                       │
│ ...                                    │
│                                        │
│ 问题：张三的季度销售额是多少？           │
└────────────────────────────────────────┘
     ↓
LLM生成: 根据检索到的内容生成答案
     ↓
输出: "根据销售报告，张三Q3销售额为..."
     ↓
引用: 附带来源文档链接（可选）
```

**RAG的关键技术点**：

| 技术                                | 作用                                   |
| ----------------------------------- | -------------------------------------- |
| **Chunking策略**              | 决定召回粒度，太大噪声多，太小信息不全 |
| **Embedding模型**             | 决定检索质量，e5/bge/bce等             |
| **Hybrid检索**                | 向量+关键词混合，兼顾语义和精确        |
| **Rerank**                    | 对召回结果重排，提高相关性             |
| **Query Rewrite**             | 改写用户Query，提升召回效果            |
| **Self-RAG / Corrective-RAG** | 检索-生成闭环校验                      |

### 2. 介绍一下LangGraph？它和LangChain的区别是什么？

**A：**

**LangGraph = LangChain团队推出的用于构建**有状态、多代理工作流**的框架**

**为什么需要LangGraph**：

| 场景需求                    | LangChain (Chains) | LangGraph           |
| --------------------------- | ------------------ | ------------------- |
| 单轮问答                    | ✅ 适合            | ✅ 也可以           |
| 多轮对话                    | ✅ 适合            | ✅ 更适合（有状态） |
| 简单流水线                  | ✅ 适合            | ✅ 可以             |
| **复杂循环工作流**    | ❌ 难以表达        | ✅ 专为循环设计     |
| **多Agent协作**       | ❌ 难以表达        | ✅ 原生支持         |
| **需要中途切换/回退** | ❌ 不支持          | ✅ 支持             |

LangChain的Chain是**有向无环图（DAG）**，无法表达循环；LangGraph是**有向有环图（DYG）**，支持Agent循环。

**LangGraph核心概念**：

```
┌────────────────────────────────────────────────────┐
│                    LangGraph                       │
├────────────────────────────────────────────────────┤
│                                                    │
│  Node（节点）= 业务逻辑的执行单元                   │
│  ├── 工具调用节点                                  │
│  ├── LLM生成节点                                   │
│  ├── 条件判断节点                                  │
│  └── 自定义函数节点                                │
│                                                    │
│  Edge（边）= 节点之间的流转关系                    │
│  ├── Normal Edge: 无条件流转                       │
│  └── Conditional Edge: 满足条件才流转             │
│                                                    │
│  State（状态）= 贯穿整个图的数据流                 │
│  ├── messages: 对话历史                            │
│  ├── context: 检索到的上下文                        │
│  └── 任何自定义字段                               │
│                                                    │
│  StateGraph = State + Nodes + Edges               │
│                                                    │
└────────────────────────────────────────────────────┘
```

**一个完整的LangGraph工作流示例**：

```
用户Query
    │
    ▼
┌─────────────────┐
│  retrieve_docs │  ──Normal Edge──▶ ┌─────────────────┐
│    (Node)       │                    │  judge_relevance│
└─────────────────┘                    │    (Node)       │
                                       └────────┬────────┘
                                                │
                           Conditional Edge     │
                        ┌──────────────────────┼──────────────────────┐
                        │ 高度相关              │  不相关               │
                        ▼                      ▼                      ▼
              ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
              │ generate_answer│     │ ask_clarification│   │ end (结束)     │
              │    (Node)      │     │    (Node)       │     │                │
              └────────┬────────┘     └────────┬────────┘     └─────────────────┘
                       │                       │
                       └───────────┬───────────┘
                                   ▼
                            ┌─────────────────┐
                            │  review_answer  │  ← 如果质量不够，重新检索
                            │    (Node)       │
                            └────────┬────────┘
                                     │
                        Conditional Edge
                        ┌────────────┴────────────┐
                        │  质量OK                  │  质量不够
                        ▼                          ▼
                  ┌─────────────┐          ┌─────────────┐
                  │    END      │          │ retrieve... │  ← 回到检索，重新生成
                  └─────────────┘          └─────────────┘
```

**State在节点间传递**：

```python
# 定义State结构
class AgentState(TypedDict):
    messages: List[BaseMessage]           # 对话历史
    context: List[Document]               # 检索到的文档
    current_node: str                     # 当前节点名
    retry_count: int                      # 重试次数

# 节点函数签名
def retrieve_docs(state: AgentState) -> AgentState:
    query = state["messages"][-1].content
    docs = vector_store.similarity_search(query, k=5)
    return {"context": docs}  # 只返回要更新的字段

def generate_answer(state: AgentState) -> AgentState:
    prompt = build_prompt(state["messages"], state["context"])
    response = llm.invoke(prompt)
    return {"messages": [...state["messages"], response]}
```

**LangGraph vs LangChain核心区别**：

| 特性                 | LangChain           | LangGraph                       |
| -------------------- | ------------------- | ------------------------------- |
| **架构**       | Chain（DAG，无环）  | Graph（有环，支持循环）         |
| **状态管理**   | 隐式，通过Chain传递 | 显式State，所有节点共享         |
| **循环支持**   | ❌ 不支持           | ✅ 支持                         |
| **多Agent**    | 难以实现            | ✅ 原生支持                     |
| **回退/重试**  | 需手动实现          | ✅ 通过Conditional Edge天然支持 |
| **可视化调试** | 困难                | ✅ 图结构清晰，易于调试         |
| **适用场景**   | 简单、线性流程      | 复杂、有状态、多轮交互          |

**一句话总结**：LangGraph = LangChain的"豪华升级版"，专门为Agent的**循环决策**而设计。

## 二、RAG与检索系统

### 3. MinerU解析出的Markdown相比纯文本，层级结构对检索有什么好处？

**A：**

- **结构保留**：标题层级（H1/H2/H3）对应文档的逻辑层次，检索时可按层级折叠或展开
- **语义边界清晰**：不同标题下的内容天然被分隔，避免跨主题的噪声召回
- **支持结构化查询**：可针对"第3章的表格"或"小结"进行精确召回
- **便于重排序**：Chunk之间的父子关系可辅助Multi-Vector RAG或Parent Document检索

### 4. VLM（视觉语言模型）是在检索阶段参与，还是只在生成答案时参与？

**A：**

- **主流范式**：VLM主要用于**生成阶段**——对召回的文档截图/表格进行视觉理解
- **检索阶段**：一般用纯文本Embedding模型（e5、bge等），视觉特征不参与召回
- **部分高级系统**：也会用VLM做**文档预索引**，例如将PPT每一页转图片后用VLM提取视觉摘要，再建索引

### 5. Ragas评测中，Faithfulness得分低说明什么问题？

**A：**

- **Faithfulness（忠诚度）**衡量的是回答与检索上下文的一致性
- **得分低 = 幻觉严重**：模型在生成时出现了**输入上下文中不存在的内容**（虚构事实、篡改数据）
- **常见原因**：
  - 检索召回的内容不足以支撑回答，模型被迫"凑答案"
  - 模型本身指令遵循能力弱，过度依赖参数记忆
  - Temperature设置过高导致随机性生成

## 二、LangGraph与Agent架构

### 6. LangGraph中Node和Edge分别代表业务流中的什么？

**A：**

- **Node（节点）**：业务流中的**处理步骤/动作**
  - 例如：`retrieve_docs`、`generate_answer`、`validate_output`
  - 每个Node是一个**可执行的函数**
- **Edge（边）**：节点之间的**流转规则/条件**
  - **Normal Edge**：无条件执行下一个节点
  - **Conditional Edge**：根据状态/输出决定走哪条路径
  - 例如：检索结果为空→结束；检索到内容→生成答案

### 7. 长短期记忆在实现上有什么区别？三级存储分别存在哪里？

**A：**

记忆系统通常分为**三级**，从近到远分别是：**会话记忆（Working Memory）→ 短期记忆（Short-Term）→ 长期记忆（Long-Term）→ 外部知识库（External KB）**

#### 三级记忆对比

|                    | 会话记忆（Working）                    | 短期记忆（Short-Term）                  | 长期记忆（Long-Term）                   | 外部知识库                        |
| ------------------ | -------------------------------------- | --------------------------------------- | --------------------------------------- | --------------------------------- |
| **内容**     | 当前对话的原始Message列表              | 压缩后的会话摘要/关键实体               | 跨会话积累的知识、用户画像              | 结构化文档、知识图谱、数据库      |
| **存储介质** | **内存**（Python dict/list变量） | **内存或本地磁盘**（SQLite/文件） | **向量数据库**（Milvus/Pinecone） | **向量数据库 + 结构化存储** |
| **生命周期** | 单次请求，会话结束销毁                 | 当前会话，会话后可选择性保留            | 持久化，跨会话                          | 持久化，定期更新                  |
| **召回方式** | 直接读取，按位置索引                   | 摘要检索/实体匹配                       | 向量相似度检索                          | 混合检索（向量+关键词）           |
| **容量**     | 受LLM上下文窗口限制（4K-200K token）   | 压缩后约数KB                            | 无限制，取决于向量库规模                | 无限制，可达TB级                  |
| **召回延迟** | <1ms（内存直接读）                     | <10ms（本地磁盘读）                     | 10-100ms（向量检索）                    | 10-500ms（远程API/数据库）        |

#### 各层级存储介质详解

**1. 会话记忆（Working Memory）—— 存在内存**

```
Python进程内存
├── messages: List[Dict]     # [{"role": "user", "content": "..."}]
├── session_id: str
└── context_window: List[str] # 滑动窗口保留的最近N轮
```

- 直接用Python list/dict存储
- 进程生命周期内有效，重启即丢失
- 也可借助Redis做进程间共享（但仍是内存级延迟）

**2. 短期记忆（Short-Term）—— 存在本地磁盘/数据库**

```
本地存储
├── SQLite: 存储会话摘要、关键实体、用户偏好
├── JSON文件: 结构化的会话存档
└── 内存缓存（Redis）: 热点短期记忆加速读取
```

- 当会话结束时，可选择性将"有价值"的内容写入短期存储
- 典型实现：SQLite文件记录每个 `session_id`对应的摘要和关键信息
- 延迟比纯内存高，但容量大，可跨进程访问

**3. 长期记忆（Long-Term）—— 存在向量数据库**

```
向量数据库（Milvus / Pinecone / Qdrant / Chroma）
├── Session Summary Embeddings: 会话摘要的向量
├── Entity Memory: 用户画像、项目经验
└── Knowledge Fact: 结构化知识事实
```

- 将信息转为Embedding存入向量库
- 通过语义相似度召回相关记忆
- 支持分布式部署，跨服务共享

**4. 外部知识库（External KB）—— 存在混合存储**

```
外部知识库
├── 向量数据库: 非结构化文档（PDF、网页、Markdown）
├── 知识图谱（Neo4j）: 实体关系三元组
├── 结构化数据库（MySQL/PG）: 表格数据、配置信息
└── 搜索引擎（Elasticsearch）: 全文检索 + 向量检索
```

- 最外层、最远距离，召回延迟最高
- 通过RAG管道综合召回多个来源

#### 分级存储的工作流程

```
用户Query
  ↓
1. 先查【会话记忆】：内存中当前session的messages（最低延迟）
  ↓ 缺失
2. 查【短期记忆】：SQLite/Redis中当前用户的近期会话摘要
  ↓ 缺失
3. 查【长期记忆】：向量库中该用户的长期记忆embedding
  ↓ 缺失
4. 查【外部知识库】：RAG检索文档/知识图谱/搜索引擎（最高延迟）
  ↓
综合所有层级的召回结果，构造成上下文 → LLM生成答案
```

#### 层级流转规则

- **会话结束时的处理**：
  - 原始Messages → **压缩为摘要** → 存入短期记忆
  - 重要实体/偏好 → 提取 → 存入长期记忆（向量库）
  - 文档/知识 → 判断是否要更新外部知识库
- **召回优先级**：按延迟从低到高依次尝试，避免每次都查外部知识库

### 8. Agent是如何识别并选择正确工具的？基于名称还是功能描述？

**A：**

- **核心机制**：基于**功能描述（description）的语义匹配**
- **具体流程**：
  1. 工具注册时提供**自然语言描述**（如"搜索2019-2024年的GDP数据"）
  2. Agent将用户Query和所有工具描述一起发给LLM
  3. LLM根据**语义相似度**判断应该调用哪个工具
  4. 工具的**名称**只是辅助标识，LLM主要依据description理解其功能
- **部分系统**：会用Embedding对工具描述建索引，实现更精准的工具召回

## 三、向量检索 vs 关键词检索

### 9. 向量检索和关键词检索各自优缺点

**A：**

|                    | 向量检索                                                     | 关键词检索（BM25/TF-IDF）            |
| ------------------ | ------------------------------------------------------------ | ------------------------------------ |
| **优点**     | 语义理解能力强；支持同义词、近义词；可处理语义相似但字面不同 | 精确匹配；可解释性强；轻量快速       |
| **缺点**     | 对专有名词/数字不敏感；需要大量向量计算；有检索精度上限      | 无法理解语义；字面一变就召回失败     |
| **适用场景** | 语义查询、"用一句话找相关内容"                               | 专有名词检索、代码搜索、精确术语匹配 |
| **召回率**   | 语义相关→高召回                                             | 字面匹配→精确召回                   |
| **排序能力** | 语义相似度排序                                               | BM25词频统计排序                     |

**工业实践**：通常**混合检索**（Hybrid Search）——向量检索+关键词检索，再用RRF或学习型重排序融合

## 四、RAG减少幻觉的原理

### 10. 为什么RAG比直接问大模型更能减少事实性错误？

**A：**

- **知识边界明确**：RAG将回答限制在检索到的上下文中，模型只在"已知范围内"生成
- **切断参数记忆依赖**：大模型的"知识"来自训练，可能过时/有错；RAG强制使用检索到的**现势性信息**
- **可溯源可验证**：答案附带引用来源，便于人工/自动化校验
- **减少模型虚构**：模型产生幻觉往往是因为"不知道答案时强行编造"，RAG提供了上下文使模型"有据可依"

## 五、向量数据库同步

### 11. 原始文档修改后，如何保证向量索引同步更新？

**A：**

- **主动同步策略**：
  1. **版本号/时间戳**：给文档和向量记录打版本号，修改时更新版本，全量/增量比对
  2. **CDC（Change Data Capture）**：监听文件系统的变更事件（inotify/notify），触发重建
  3. **MD5哈希比对**：定期计算文档哈希，与上次索引时的哈希对比，变了则重建
- **软删除+重建**：不直接删向量，而是标记 `deleted=True`，查询时过滤；后台异步重建
- **增量索引**：只重建变化的Chunk，不影响其他已索引内容

## 六、Query Rewrite

### 12. 什么是Query Rewrite？能解决什么问题？

**A：**

- **定义**：对用户原始Query进行改写/扩展/纠错后再检索
- **解决的问题**：
  - **口语转书面语**：用户说"咋回事"→"发生了什么"
  - **缩写/别名**："RAG"→"Retrieval Augmented Generation"
  - **query扩展**：短query补充上下文，变成完整问句
  - **拼写纠错**：自动纠正用户输入错误
  - **同义词改写**：用领域词典替换为标准术语
- **实现方式**：可以用小模型做rewrite，或用规则/词典

## 七、Temperature参数

### 13. Temperature调高和调低分别对输出有什么影响？

**A：**

- **Temperature → 0（低）**：输出**确定性强、重复性低**
  - 模型更倾向选择概率最高的token
  - 适合任务：代码生成、数学计算、事实问答
- **Temperature → 1（高）**：输出**多样性高、随机性强**
  - 模型会采样更多低概率token
  - 适合任务：创意写作、头脑风暴、角色扮演
- **Temperature > 1（极高）**：输出近乎随机，可能出现语法混乱
- **本质**：Temperature是对softmax logits的缩放，`p_i = softmax(logit_i / T)`

## 八、Chain of Thought（CoT）

### 14. 讲一下CoT的原理？为什么它能让模型处理复杂任务？

**A：**

- **核心思想**：让模型显式输出推理步骤，而非直接给出答案
- **原理**：
  1. 复杂任务需要**多步推理**，每步推理依赖前一步结果
  2. 直接生成答案时，模型"压缩"了推理过程，容易出错
  3. CoT让模型**将推理过程展开**，每一步都经过显式计算
  4. 推理步骤形成"中间变量"，降低了每步的计算复杂度
- **为什么有效**：
  - 增加了token序列长度，提供更多"计算空间"
  - 推理步骤本身有自检性，模型能发现早期错误
  - 激活了模型在预训练中学到的逻辑推理模式
- **变种**：Zero-shot CoT（"Let me think step by step"）、Few-shot CoT（给示例）、Self-Consistency

## 九、Agent循环机制

### 15. Thought、Action和Observation三者如何循环？

**A：**

```
User Query → Thought → Action → Observation → Thought → Action → Observation → ...
                                              ↓
                                         (满足终止条件)
                                              ↓
                                          Final Answer
```

- **Thought**：模型分析当前状态，决定"要做什么"
- **Action**：执行具体操作（查资料、调用工具、写代码）
- **Observation**：接收Action的返回结果，更新上下文
- **循环终止条件**：
  - 模型判断任务完成
  - 达到最大循环次数
  - 出现异常/超时

## 十三、HNSW索引

### 19. 什么是HNSW索引？为什么比暴力搜索快？

**A：**

- **HNSW（Hierarchical Navigable Small World）**：一种基于图的多层近似最近邻搜索算法
- **核心思想**：
  1. 构建多层图，上层稀疏、下层稠密
  2. 搜索时从最上层开始，按贪婪游走找到最近邻的大致位置
  3. 逐层下沉，最终在下层精确定位
- **为什么快**：
  - **剪枝**：上层节点稀疏，大多数节点在早期就被排除
  - **小世界跳跃**：同层内可快速跳跃到远距离节点，避免遍历全图
  - **复杂度**：从O(N)降到O(log N)（近似）
- **对比**：
  - 暴力搜索：O(N)，遍历所有向量
  - HNSW：O(log N)，只搜索部分节点
  - IVF-PQ：倒排索引+量化，进一步加速

## 十四、Transformer 基础（八股）

### 20. Transformer整体架构？它解决了RNN和CNN哪些无法解决的问题？

**A：**

**Transformer整体架构**：

```
输入 Embedding + 位置编码
       ↓
┌─────────────────────────┐
│   Encoder Stack (N层)    │
│  Multi-Head Self-Attention │
│         +               │
│   Feed Forward Network  │
│         +               │
│      LayerNorm          │
│     残差连接            │
└─────────────────────────┘
       ↓
┌─────────────────────────┐
│   Decoder Stack (N层)   │
│  Masked Self-Attention  │
│         +               │
│   Cross-Attention       │
│   (Encoder-Decoder)     │
│         +               │
│   Feed Forward Network  │
│         +               │
│      LayerNorm          │
│     残差连接            │
└─────────────────────────┘
       ↓
     Linear + Softmax
       ↓
      输出概率
```

**相比RNN的优势**：

| 问题                 | RNN                                 | Transformer                 |
| -------------------- | ----------------------------------- | --------------------------- |
| **并行计算**   | 必须串行（hidden依赖）              | 完全并行，GPU友好           |
| **长距离依赖** | 梯度消失/爆炸，理论上可处理，实际难 | Self-Attention O(1)路径长度 |
| **信息衰减**   | 每步衰减，越远越难捕获              | 任意两个位置直接交互        |
| **训练效率**   | 低                                  | 高（矩阵并行）              |

**相比CNN的优势**：

| 问题                   | CNN                                | Transformer             |
| ---------------------- | ---------------------------------- | ----------------------- |
| **感受野**       | 固定局部，需多层堆叠才能捕获远距离 | 全局注意力一步到位      |
| **位置建模**     | 隐式（通过卷积核），需大量filter   | 显式位置编码            |
| **自适应感受野** | 固定（kernel size）                | Query-Key-Value动态计算 |

**核心问题Transformer无法解决**：没有解决——参数量大、推理慢、需要大量训练数据

### 21. 讲一下QKV的原理？它的核心作用是什么？

**A：**

**QKV定义**：

```python
Q = X @ W_q   # Query: "我需要什么信息"
K = X @ W_k   # Key: "我包含什么信息"
V = X @ W_v   # Value: "信息的实际内容"
```

- 输入X是token的embedding向量（假设768维）
- 三个权重矩阵W_q/W_k/W_v是可学习的（假设输出也是768维）

**核心作用**：

| 矩阵        | 本质     | 类比             |
| ----------- | -------- | ---------------- |
| **Q** | 查询向量 | "我在找什么"     |
| **K** | 键向量   | "我能提供什么"   |
| **V** | 值向量   | "实际传递的内容" |

**Attention计算过程**（以单个Query为例）：

```
1. 计算单个Query对所有Key的相似度：
   score = Q @ K.T / sqrt(d_k)   # shape: (1, seq_len)

2. Softmax归一化得到注意力权重：
   attention_weights = softmax(score)  # shape: (1, seq_len)

3. 用注意力权重对Value加权求和：
   output = attention_weights @ V     # shape: (1, d_v)
```

**为什么要除以sqrt(d_k)**：防止点积结果过大，导致softmax进入饱和区（梯度消失）

**Multi-Head Attention**：将QKV分成h个头，各自独立计算Attention，再拼接：

```python
# 假设 d_model=512, num_heads=8, d_k=64
for h in range(8):
    Q_h = X @ W_q[h]   # (batch, seq, 64)
    K_h = X @ W_k[h]   # (batch, seq, 64)
    V_h = X @ W_v[h]   # (batch, seq, 64)
    head_h = attention(Q_h, K_h, V_h)  # 各自计算

concat(head_0, ..., head_7) @ W_o  # 拼接后线性变换
```

**为什么需要Multi-Head**：

- 不同的头可以关注不同的语义子空间（语法、实体、关系等）
- 提供模型的表达能力，比单头更robust
- 类似于卷积中多个filter的作用

### 22. Transformer的核心机制是什么？用一个具体例子串一遍

**A：**

以句子 **"The cat sat on the mat because it was tired"** 为例：

**Step 1: 输入编码**

```
Token序列: [The, cat, sat, on, the, mat, because, it, was, tired]
     ↓
Embedding: 每个token转为768维向量
     ↓
+ Position Encoding: 加入位置信息
     ↓
输入矩阵 X: (10, 768)  # 10个token，768维
```

**Step 2: Encoder自注意力（以"it"为例）**

```
"it"的Query "it"去查询所有token的Key：
- vs "cat": score=0.8 (高，"it"指代"cat")
- vs "tired": score=0.7 (高，"it"的状态)
- vs "mat": score=0.1 (低，不相关)

注意力权重: [0.02, 0.35, 0.01, 0.01, 0.02, 0.05, 0.01, 0.35, 0.08, 0.10]
 → "it"的向量主要聚合了"cat"和"tired"的信息
```

**Step 3: Multi-Head各头关注不同方面**

```
Head 0 (语法): "it"主要看动词"was"，捕捉主谓关系
Head 1 (指代消解): "it"→"cat"，实体链接
Head 2 (情感): "tired"→负面情绪
...
```

**Step 4: FFN前馈网络**

- 对每个位置的输出做非线性变换
- 两层全连接 + ReLU/GELU
- 进一步抽象特征

**Step 5: Decoder自回归生成**

```
输入: "<bos> The cat sat on the"
     ↓
Decoder Masked Attention: "it"只能看前面的token
     ↓
Cross Attention: 用Encoder输出作为K和V
     ↓
输出: " because it was tired <eos>"
```

**整个流程中的关键操作**：

| 操作             | 作用                   | 位置            |
| ---------------- | ---------------------- | --------------- |
| Self-Attention   | 任意位置交互，捕获依赖 | Encoder/Decoder |
| Masked Attention | 防止看到未来信息       | Decoder         |
| Cross Attention  | Decoder利用Encoder信息 | Decoder         |
| FFN              | 非线性特征变换         | Encoder/Decoder |
| 残差连接         | 梯度流动，保留原始信息 | 每个子层后      |
| LayerNorm        | 稳定训练，加速收敛     | 每个子层后      |

### 23. 模型层叠后会出什么问题？为什么要引入残差和LayerNorm？

**A：**

**层叠后的问题**：

1. **梯度消失/爆炸**：深层网络反向传播时，梯度连乘导致不稳定
2. **特征分布偏移**：每层输入分布随网络加深而变化（Internal Covariate Shift）
3. **训练困难**：深层网络难以收敛，需要仔细初始化

**残差连接（Residual Connection）**：

```python
output = LayerNorm(x + Sublayer(x))
# Sublayer是注意力或FFN子层
```

**为什么有效**：

- **梯度高速公路**：反向传播时，梯度可以直接从输出传到输入（`x + ...`的路径上梯度始终为1）
- **恒等映射**：如果子层学到的是恒等变换，网络退化为浅层，不会有副作用
- **缓解梯度消失**：即使子层梯度很小，`x +`那条路径保证信息能传下去

**LayerNorm**：

```python
# 对每个样本的所有特征做归一化
x = (x - mean(x)) / std(x)
x = x * gamma + beta  # 可学习的缩放和偏移
```

**为什么有效**：

- **稳定梯度**：将 activations 拉到均值为0、方差为1的分布
- **加速收敛**：让每层的输入分布相对稳定
- **正则化效果**：减少 ICS（Internal Covariate Shift）

**Pre-LN vs Post-LN**：

|            | Post-LN（原始）          | Pre-LN（主流）           |
| ---------- | ------------------------ | ------------------------ |
| 结构       | `x + LN(Sublayer(x))`  | `x + Sublayer(LN(x))`  |
| 梯度       | LN在路径上，梯度可能消失 | 残差路径无LN，梯度更顺畅 |
| 训练稳定性 | 较差，需要warm-up        | 更稳定                   |
| 收敛速度   | 慢                       | 快                       |

**现代LLM几乎都使用Pre-LN**

### 24. 传统残差机制有什么问题？业界有没有提出新的解决方案？

**A：**

**传统残差的问题**：

1. **信息瓶颈**：F(x) = x + Sublayer(x)，如果Sublayer输出与x分布差异大，加法会丢失细粒度信息
2. **带宽受限**：深层网络中每层都要"记住"前面的所有信息，容量有限
3. **无法选择性遗忘**：没有机制让网络主动决定哪些信息可以丢弃

**新方案**：

**1. Pre-LN + 归一化改进**

- Pre-LN将LayerNorm移到残差分支内部，梯度更顺畅（已广泛采用）

**2. RMSNorm（Root Mean Square Norm）**

```python
# LayerNorm: 减均值 + 除标准差
# RMSNorm: 只除RMS（更快，忽略均值）
RMSNorm(x) = x / RMS(x) * gamma
# T5、GLM等模型使用，性能相当但快30-40%
```

**3. DeepNorm（微软）**

```python
output = LayerNorm(alpha * x + Sublayer(x))
# alpha > 1，放大残差路径，用于极深网络（如2层~100+层）
# Google PaLM使用
```

**4. 门控残差（Gated Residual）**

```python
output = x + gate * Sublayer(x)
# 门控机制让网络自己学习残差的权重
# 像是一个"信息过滤器"
```

**5. SkipCo（Skip Connections via Compression）**

- 对残差路径做降维/压缩，减少信息传输压力

**6. 苏剑林提出的"用FFN残差代替注意力残差"**

- 认为主要信息流应该走FFN路径而非注意力路径
- 实验显示FFN比注意力更需要残差连接

**总结**：残差机制的核心思想已确定（信息高速公路），主要改进在**归一化位置/方式**和**门控机制**上

### 25. "距离"的概念在Transformer中是怎么体现的？为什么要知道词与词之间的距离？

**A：**

**位置编码（Position Encoding）**——显式建模距离：

Transformer通过正弦/余弦位置编码让模型"知道"token之间的距离：

```python
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

- 位置pos相同距离为0
- 差值代表相对距离
- 任意两个位置的距离可以用内积计算

**Attention机制——隐式捕获距离**：

```python
# Attention score计算中隐含距离信息
score(q_i, k_j) = q_i @ k_j.T / sqrt(d_k)
# 这个score没有显式加入距离，但...
```

实际上Attention是**位置感知**的，因为：

- QKV来自带位置编码的输入
- 注意力权重天然反映了token之间的关联强度（不管距离）

**为什么要知道距离**：

1. **语序建模**："狗咬人" vs "人咬狗"——距离决定主客体关系
2. **指代消解**："它"指代哪个词——需要知道"它"和哪些词更近
3. **局部语法**：形容词修饰名词（紧邻），动词和主语（可远）
4. **长距离依赖**：跨句、跨段的关系需要明确捕获

**为什么仅靠Attention不够（需要位置编码）**：

- **Permutation Invariance**：Attention对输入顺序不敏感
- 打乱token顺序，注意力分数不变 → 输出不变
- 位置编码打破了这种不变性，让模型能区分"第1个词"和"第2个词"

**现代模型的改进**：

| 模型            | 位置编码方式             |
| --------------- | ------------------------ |
| Transformer原始 | 正弦/余弦（绝对位置）    |
| BERT            | 可学习的绝对位置编码     |
| RoPE（Llama等） | 旋转位置编码，相对位置   |
| ALiBi           | 线性偏置，不需要位置编码 |
| 相对位置编码    | 显式建模相对距离         |

**RoPE为什么流行**：同时编码绝对位置（通过旋转）和相对位置（通过内积性质），且可外推到更长序列

### 26. Transformer的核心参数有哪些？b是什么概念？

**A：**

**b = billion（十亿）**：70B = 700亿参数

**Transformer核心参数分布**：

| 组件                           | 参数量公式                      | 70B模型中的占比   |
| ------------------------------ | ------------------------------- | ----------------- |
| **Embedding & 位置编码** | V × d + d                      | ~0.1%             |
| **QKV投影**              | 3 × d²（每个Attention头）     | ~5%               |
| **Output投影**           | d²                             | ~1%               |
| **FFN**                  | 4 × d²（两个线性层）          | **~60-70%** |
| **LayerNorm**            | 2 × d                          | ~<0.1%            |
| **多头总览**             | h × 4 × d²（Multi-Head汇总） | 包含在FFN中       |

**FFN为什么占比最高**：

```python
# FFN通常是
x -> W1 (d, 4d) -> gelu -> W2 (4d, d)
# 参数量: d × 4d + 4d × d = 8d²
# Attention: QKV各d² + O = 4d²
# FFN是Attention的2倍
```

**以70B、d=8192、h=64为例**：

```
d_model = 8192
num_heads = 64
d_head = 128

Embedding: ~1M
每个Attention层: 64 × (3×128² + 128²) ≈ 6.5M
每个FFN层: 2 × (8192 × 32768) ≈ 536M
总参数量 ≈ 70B
```

**为什么选7B/32B/70B**：

| 参数量          | 典型场景           | 显存（FP16） | 能力                |
| --------------- | ------------------ | ------------ | ------------------- |
| **7B**    | 本地部署、边缘设备 | ~14GB        | 基础对话、简单Agent |
| **13B**   | 中等复杂度任务     | ~26GB        | 较好推理、工具调用  |
| **32B**   | 复杂推理、多轮对话 | ~64GB        | 较强指令遵循        |
| **70B**   | 复杂规划、代码生成 | ~140GB       | 接近GPT-4水平       |
| **100B+** | 学术研究、AGI探索  | >200GB       | 前沿能力探索        |

**选择依据**：

- 任务复杂度决定最低参数量
- 硬件预算决定最高参数量
- 在两者之间选性价比最高的

## 十五、模型选型

### 27. 开发Agent应用时，怎么判断用7B小模型还是70B大模型？

**A：**

- **7B适用场景**：
  - 简单、单一任务（如分类、摘要、格式转换）
  - 对延迟敏感（延迟低，成本低）
  - 本地部署/边缘设备
  - 有明确规则的场景
- **70B适用场景**：
  - 复杂推理任务（多步规划、代码生成）
  - 工具调用频繁且多样
  - 对话理解要求高
  - 需要强指令遵循能力
- **判断原则**：
  - 任务复杂度 = 决定性因素
  - 延迟要求 vs 效果要求
  - 成本预算
  - 是否需要多语言/长上下文

### 28. 如果Agent总是重复执行同一个错误的工具调用，从哪个环节修复？

```
Tool descriptions → LLM理解 → Tool selection → Execution → Result parsing
       ↑                ↑              ↑             ↑            ↑
    (问题可能在这里)
```

- **可能原因及修复**：

| 问题环节                 | 症状                  | 修复方向                                |
| ------------------------ | --------------------- | --------------------------------------- |
| **Tool描述不清**   | LLM总是选错工具       | 优化description，加入更多示例和边界说明 |

## 十七、MCP（Model Context Protocol）

### 29. 什么是MCP？它解决了什么问题？

**A：**

**MCP = Model Context Protocol（模型上下文协议）**

MCP是一种用于连接AI模型与外部工具、数据源的标准协议，由Anthropic在2024年提出（类似USB之于硬件设备的角色）。

**解决的问题**：

| 问题                 | 没有MCP                                 | 有MCP                                  |
| -------------------- | --------------------------------------- | -------------------------------------- |
| **工具接入**   | 每个Agent框架自己定义工具接口，互不兼容 | 统一协议，一次开发，任何MCP Client可用 |
| **数据源连接** | 每个LLM应用自己实现                     | MCP Server提供标准化的数据访问         |
| **重复开发**   | Cursor/Cline/Claude Desktop各自实现一套 | 一次实现，处处运行                     |
| **安全边界**   | 工具权限不清晰                          | MCP定义明确的权限作用域                |

**MCP架构**：

```
┌─────────────────────────────────────────────────────────┐
│                      Host App                           │
│   (Claude Desktop / Cursor / Cline / 其他AI应用)        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐ │
│  │ MCP Client  │    │ MCP Client  │    │ MCP Client  │ │
│  │   (stdio)   │    │  (HTTP/SSE) │    │  (WebSocket)│ │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘ │
│         │                  │                   │        │
│  ┌──────┴──────────────────┴───────────────────┴──────┐ │
│  │                   MCP Protocol                     │ │
│  │  JSON-RPC 2.0 + 标准化Schema                      │ │
│  └──────┬──────────────────┬───────────────────────┬────┘ │
│         │                  │                       │       │
│  ┌──────┴──────┐    ┌──────┴──────┐    ┌────────┴─────┐ │
│  │ File System │    │   Database  │    │  Web Search  │ │
│  │   Server    │    │   Server    │    │    Server    │ │
│  └─────────────┘    └─────────────┘    └──────────────┘ │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心价值**：MCP就像AI世界的"USB接口"，让工具和数据源的接入变得即插即用。

### 30. MCP的常见协议有哪些？它们的区别是什么？

**A：**

MCP支持多种传输协议，用于Client与Server之间的通信：

#### 1. stdio（标准输入/输出）

**原理**：通过进程的标准输入输出进行JSON-RPC通信

```
Host进程 ──stdin──▶ MCP Server
Host进程 ◀──stdout── MCP Server
```

**配置示例**：

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/dir"]
    }
  }
}
```

| 优点       | 缺点                   |
| ---------- | ---------------------- |
| 简单易实现 | 只能本地进程，无法远程 |
| 低延迟     | 无法跨机器通信         |
| 无网络依赖 | 难以水平扩展           |

**适用场景**：本地工具服务、Cursor/Cline插件

---

#### 2. HTTP + SSE（Server-Sent Events）

**原理**：Client通过HTTP POST发送请求，Server通过SSE推送响应

```
Client ──POST /mcp─────▶ Server (请求)
Client ◀──SSE /events── Server (响应流)
```

**特点**：

- 支持长连接
- Server可以主动推送
- 适合实时性要求高的场景

| 优点           | 缺点             |
| -------------- | ---------------- |
| 支持服务端推送 | 需要HTTP基础设施 |
| 可远程访问     | 防火墙配置复杂   |
| 适合流式响应   | 比stdio延迟略高  |

**适用场景**：Web应用、远程工具服务

---

#### 3. WebSocket

**原理**：全双工通信通道，Client和Server可以互相主动发送消息

```
Client ⟺ Server (双向实时通信)
```

**特点**：

- 真正的全双工
- 连接建立后Server可主动推送
- 支持二进制帧

| 优点       | 缺点                   |
| ---------- | ---------------------- |
| 最低延迟   | 实现复杂度高           |
| 全双工通信 | 需要WebSocket服务器    |
| 连接复用   | Proxy/负载均衡配置复杂 |

**适用场景**：高频交互场景、实时协作应用

---

#### 4. Streamable HTTP

**原理**：MCP官方推荐的现代传输方式，支持多种HTTP特性

```
Client ──POST /message──▶ Server
Client ◀──200 OK + body── Server
```

**特点**：

- 统一的HTTP端点
- 支持分块传输
- 可升级为WebSocket
- 默认使用JSON-RPC 2.0

---

#### 协议对比总结

| 协议                      | 实时性 | 远程支持 | 复杂度 | 适用场景              |
| ------------------------- | ------ | -------- | ------ | --------------------- |
| **stdio**           | ★★★ | ❌       | ★     | 本地工具、快速原型    |
| **HTTP+SSE**        | ★★☆ | ✅       | ★★   | Web应用、简单远程服务 |
| **WebSocket**       | ★★★ | ✅       | ★★★ | 高频交互、实时协作    |
| **Streamable HTTP** | ★★☆ | ✅       | ★★   | 通用场景、现代MCP推荐 |

**实际选择**：

- 本地插件/Desktop应用 → stdio
- 需要远程 → HTTP/SSE或Streamable HTTP
- 高频实时交互 → WebSocket

### 31. MCP Server是如何暴露工具的？工具调用的完整流程是什么？

**A：**

#### MCP Server工具暴露机制

MCP Server通过**标准化的JSON Schema**声明工具：

```python
# Python MCP Server示例
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("My Server")

@mcp.tool(description="搜索网络获取最新信息")
def web_search(query: str, top_k: int = 5) -> list:
    """搜索并返回结果"""
    results = search_engine.query(query, top_k)
    return results
```

**暴露的元数据**：

```json
{
  "name": "web_search",
  "description": "搜索网络获取最新信息",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {"type": "string"},
      "top_k": {"type": "integer", "default": 5}
    },
    "required": ["query"]
  }
}
```

#### 完整调用流程

```
1. Host启动 → 加载MCP Server配置
        ↓
2. Server注册工具 → 发送tools/list给Client
        ↓
3. LLM决定调用工具 → Client发送tools/call
        ↓
4. Server执行工具 → 返回执行结果
        ↓
5. Client将结果反馈给LLM → LLM生成最终回答
```

**具体消息流**：

```
Client                              Server
  │                                    │
  │ ── tools/list ──────────────────▶ │  请求可用工具列表
  │ ◀─ tools/list-result ──────────── │  返回工具Schema
  │                                    │
  │ ── tools/call ───────────────────▶ │  调用web_search
  │   {                                │
  │     name: "web_search",            │
  │     arguments: {query: "..."}      │
  │   }                                │
  │ ◀─ tools/call-result ───────────── │  返回搜索结果
  │                                    │
```

#### MCP资源（Resources）机制

除了工具，MCP还支持**资源**——供LLM读取的数据：

```python
@mcp.resource("docs://architecture")
def get_architecture_doc() -> str:
    """返回架构文档内容"""
    return open("architecture.md").read()
```

**Resource vs Tool**：

|         | Resource             | Tool                 |
| ------- | -------------------- | -------------------- |
| 操作    | 读取                 | 执行/写入            |
| 副作用  | 无                   | 可能有               |
| 示例    | 文件内容、数据库查询 | 搜索、发邮件、写文件 |
| LLM用途 | 提供上下文           | 执行动作             |

### 32. MCP和Function Calling有什么区别？为什么要用MCP？

**A：**

#### 核心区别

| 维度                 | Function Calling     | MCP                        |
| -------------------- | -------------------- | -------------------------- |
| **标准化程度** | 各家LLM API自己定义  | 统一协议，跨平台           |
| **工具发现**   | 每次请求传递工具定义 | Server声明，Client缓存     |
| **传输协议**   | 仅HTTP/REST          | stdio/HTTP/SSE/WebSocket   |
| **工具生态**   | 无生态，各自实现     | 丰富的预构建Server生态     |
| **权限控制**   | 无标准               | 明确的资源权限作用域       |
| **使用场景**   | LLM API调用工具      | Agent与各种数据源/工具连接 |

#### Function Calling示例

```json
// OpenAI Function Calling
{
  "functions": [{
    "name": "search",
    "description": "搜索网络",
    "parameters": {
      "type": "object",
      "properties": {
        "query": {"type": "string"}
      }
    }
  }]
}
```

#### MCP示例

```json
// MCP工具定义（标准化 Schema）
{
  "tools": [{
    "name": "search",
    "description": "搜索网络",
    "inputSchema": {
      "type": "object",
      "properties": {
        "query": {"type": "string"}
      }
    }
  }]
}
```

#### 为什么需要MCP？

1. **生态价值**：不用重复造轮子，有现成的MCP Server可用

   - 文件系统、Git、Bash
   - 数据库、API服务
   - Slack、Discord等消息应用
2. **安全隔离**：MCP Server运行在独立进程，权限边界清晰
3. **协议优势**：支持多种传输方式，适配不同场景
4. **互操作性**：Cursor/Cline/Claude Desktop共用同一套协议

**一句话总结**：Function Calling是LLM API的扩展能力，MCP是Agent与外部世界交互的标准接口。

## 十八、行程规划Agent设计

### 33. 如何设计一个行程规划Agent？有哪些核心模块？如何预留扩展能力？

**A：**

#### 核心模块设计

一个完整的行程规划Agent通常包含以下核心模块：

```
┌─────────────────────────────────────────────────────────────┐
│                    行程规划 Agent 架构                       │

│  │        → 意图: travel_plan, 槽位: {city:"北京",       │  │
│  │           days:3, budget:5000, date:"下周"}           │  │
│  └─────────────────────────────────────────────────────┘  │

│  │          2. Knowledge Retrieval（知识检索）          │  │

│                           ↓                                 │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           4. External Tools（外部工具）              │  │

│  └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 核心模块详解

**1. Intent Understanding（意图理解）**

```python
# 槽位填充示例
@dataclass
class TravelSlot:
    destination: str          # 目的地
    departure: str            # 出发地
    date_start: Optional[str]  # 开始日期
    date_end: Optional[str]    # 结束日期
    days: Optional[int]       # 天数
    budget: Optional[float]   # 预算
    travelers: int            # 出行人数
    preferences: List[str]    # 偏好标签 ["亲子", "美食", "购物"]
    transportation: str       # 交通方式偏好 ["飞机", "高铁", "自驾"]
```

**意图分类 + 槽位填充的Pipeline**：

```
用户Query → LLM意图分类 → 槽位提取 → 缺省值补全 → 确认/澄清
```

**2. Knowledge Retrieval（知识检索）**

```
检索内容分层：
├── 静态知识（目的地基础信息）
│   ├── 景点库：排名、开放时间、门票、简介
│   ├── 美食库：特色餐厅、小吃、必吃榜
│   └── 交通指南：机场/火车站到市区、各景点间交通
│
├── 动态知识（实时信息，需API）
│   ├── 实时天气（影响景点选择）
│   ├── 酒店价格/空房
│   ├── 机票/火车票余量
│   └── 景点预约情况
│
└── 用户知识（个性化）
    ├── 历史偏好（去过哪里、喜欢什么类型）
    └── 特殊需求（老人/儿童/宠物）
```

**3. Planning Engine（规划引擎）**

| 子模块                      | 职责               | 技术方案                        |
| --------------------------- | ------------------ | ------------------------------- |
| **Day Scheduler**     | 把景点分配到每天   | 贪心分配 / 强化学习             |
| **Route Optimizer**   | 每天内景点顺序优化 | 贪心 + 2-opt局部搜索 / OR-Tools |
| **Budget Calculator** | 费用估算与超支预警 | 统计 + 规则                     |
| **Conflict Detector** | 检测时间/空间冲突  | 规则引擎                        |

```python
# 路线优化伪代码
def optimize_route(pois: List[POI], start: Location, end: Location) -> List[POI]:
    # 1. 按地理位置粗排序（减少绕路）
    sorted_pois = cluster_and_sort_by_location(pois)
    # 2. 计算时间窗约束（开放时间、最晚入园时间）
    valid_pois = filter_by_time_windows(sorted_pois, daily_time_budget=9h)
    # 3. 贪心 + 局部优化
    route = greedy_insert(valid_pois)
    route = two_opt_improve(route)  # 交换法优化
    return route
```

**4. External Tools（外部工具）**

通过MCP或Function Calling接入：

```python
}
```

**5. Response Generator（响应生成）**

```
输出格式支持：
├── 卡片式行程单（结构化、便于复制）
├── 地图可视化（标记所有景点）
├── 日程表视图（按时间轴展示）
├── 费用明细表（分项列出每笔费用）
└── 一键预订（直接跳转OTA下单）
```

#### 扩展能力设计（关键面试点）

**核心原则**：模块化 + 配置驱动 + 插件化

##### 1. 工具扩展：Plugin + MCP架构

```
新增工具（如"签证办理"）时，只需要：
1. 实现新的MCP Server
2. 在配置文件中注册
3. 无需修改核心代码
```

```yaml
# config/tools.yaml
enabled_tools:
  - flight_search      # 机票
  - hotel_search       # 酒店
  - attraction_ticket  # 门票
  - restaurant_reserve # 餐厅预订
  - visa_service       # 签证 ← 新增只需改配置
```

**设计模式**：**策略模式** + **工厂模式**

- 每个工具是独立的策略类

## 二十、RAG 漏召与检索优化

### 35. RAG 出现漏召时，如何从检索模块入手优化？

**A：**

**漏召（Recall Failure）** = 正确答案不在召回结果中，用户得不到满意答案。

#### 漏召原因分析框架

```
用户Query
    ↓
┌─────────────────────────────────────────────────────┐
│              漏召可能发生的环节                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. Query理解层                                      │
│     ├── Query写得太差（口语化、缺主语）              │
│     ├── 专有名词/缩写未改写（"北漂"不知指北京）      │
│     └── 同义词未扩展（"手机"vs"智能手机"）           │
│                                                     │
│  2. Embedding编码层                                  │
│     ├── Embedding模型不匹配领域（通用模型训金融文档） │
│     ├── Query和Doc编码空间不一致                     │
│     └── 句子太长/太短，信息密度低                    │
│                                                     │
│  3. 索引层                                           │
│     ├── 分块策略不合理（跨段落切断语义）            │
│     ├── Chunk太小（缺少上下文）或太大（噪声多）     │
│     └── 索引结构问题（HNSW参数没调好）               │
│                                                     │
│  4. 检索策略层                                       │
│     ├── 单一向量检索不够（缺乏关键词精确匹配）        │
│     ├── Top-K设置不当（K太小或太大）                │
│     └── 缺少混合检索（向量+BM25融合）               │
│                                                     │
│  5. 后处理层                                         │
│     ├── 缺少重排序（精排前的粗排质量不够）           │
│     └── 相似度阈值过滤不当                           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

#### 优化方案（按环节）

**1. Query改写（Query Rewrite）**

```python
# 多路Query改写，增加召回路径
def query_rewrite(query: str) -> List[str]:
    rewrites = [
        query,                        # 原句
        expand_abbreviation(query),   # 缩写展开
        expand_synonym(query),        # 同义词扩展
        rewrite_to_formal(query),     # 口语→书面语
        decompose_compound(query),   # 拆解复合句
    ]
    return rewrites
```

|----------|----------|------|

| 同义词扩展 | 口语/书面语混合 | 词向量找相似词 |
| Query分解 | 复合问句 | 按语义切分多个子查询 |
| 假设性改写 | 缺主语/指代不清 | HyDE（用LLM生成假设性答案再检索） |

**5. 检索策略优化（混合检索 + 多路召回）**

```python
# 多路召回 + RRF融合
def hybrid_search(query: str, top_k: int = 20):
    # 向量检索
    vector_results = vector_store.search(query, k=top_k*2)

    # 关键词检索（BM25）
    bm25_results = bm25_index.search(query, k=top_k*2)

    # 稀疏检索（TF-IDF）
    sparse_results = sparse_index.search(query, k=top_k*2)

    # RRF融合（Reciprocal Rank Fusion）
    fused = rrf_fusion([
        (vector_results, 0.6),    # 向量权重
        (bm25_results, 0.3),     # BM25权重
        (sparse_results, 0.1),   # 稀疏权重
    ], k=60)

    return fused[:top_k]

# RRF公式
def rrf(results_list, k=60):
    scores = defaultdict(float)
    for results, weight in results_list:
        for rank, doc in enumerate(results):
            scores[doc.id] += weight / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: -x[1])
```

**6. 后处理优化（重排序 + 过滤）**

```python
# 两阶段检索：粗召回 → 精排
# Stage 1: 向量/HNSW召回Top-100
coarse_results = vector_store.search(query, k=100)

# Stage 2: Cross-Encoder重排序
reranker = load_reranker("cross-encoder/ms-marco")
reranked = reranker.rank(query, [doc.text for doc in coarse_results])

# 输出Top-10
return reranked[:10]
```

**召回率 vs 精度平衡策略**：

| 场景 | 目标 | Top-K | 阈值 | 策略 |
|------|------|-------|------|------|
| 搜索 | 召回为主 | 100 | 0.5 | 增大K、降低阈值、混合检索 |
| 问答 | 精度为主 | 10 | 0.7 | 重排序、严格过滤 |
| 摘要 | 平衡 | 30 | 0.6 | 多路召回 + 阈值过滤 |

### 37. RAG 中检索模块召回错误结果时如何处理？如何判断是检索问题还是生成问题？

**A：**

#### 判断方法——隔离测试

```
测试1：固定检索内容，换不同生成模型
  → 答案仍错 → 问题在检索
  → 答案变好 → 问题在生成

测试2：固定生成模型，换不同检索内容
  → 换了好检索后答案正确 → 检索问题
  → 换了仍错 → 生成问题
```

| 辅助指标 | 含义 | 判断 |
|----------|------|------|
| **Retriever Precision@1** | Top-1结果相关性 | Top-1不对 → 检索问题严重 |
| **Context Precision** | 召回内容中有多少相关 | 低 → 检索噪声多 |
| **Faithfulness** | 生成与召回内容一致性 | 低 + 检索对 → 生成问题 |

#### 检索问题处理

- 换更好的Embedding模型
- 调整Top-K
- 加入重排序
- 优化分块策略

#### 生成问题处理

- 强化Prompt约束（"必须基于上下文回答"）
- 降低Temperature（数值问答要稳定）
- 必要时要求模型输出"文中未提及"

### 38. 做过哪些 rerank 或过滤策略？如何平衡召回率和精度？

**A：**

#### Rerank策略

**1. Cross-Encoder Rerank**

```python
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query: str, docs: List[str], top_k: int = 10):
    pairs = [(query, doc) for doc in docs]
    scores = reranker.predict(pairs)
    reranked = sorted(zip(docs, scores), key=lambda x: -x[1])
    return reranked[:top_k]
```

| 优点 | 缺点 |
|------|------|
| 精度高（深度交互） | 慢（无法预计算） |
| 效果明显 | 不能用于粗排（候选太多） |

**2. 多阶段Rerank流程**

```
Stage 1：向量检索（HNSW）→ 召回Top-100
Stage 2：BM25检索 → 补充Top-50
Stage 3：RRF融合 → Top-60
Stage 4：Cross-Encoder精排 → Top-10
```

#### 过滤策略

```python
# 1. 相似度阈值过滤
filtered = [doc for doc in results if doc.score > threshold]

# 2. 去重过滤（MD5）
# 3. 长度过滤（min_len, max_len）
# 4. 来源加权过滤（信任源 × 权重）
```

#### 平衡策略

| 目标 | Top-K | 阈值 | 混合检索 | Rerank |
|------|-------|------|----------|--------|
| **高召回** | 大（100） | 低 | 开启 | 后置 |
| **高精度** | 小（10） | 高 | 开启 | 必做 |
| **平衡** | 中（20-50） | 中 | 开启 | 轻量 |

## 二十三、大模型训练与优化

### 40. 解释 LoRA 原理，为什么用低秩分解？

**A：**

#### LoRA（Low-Rank Adaptation）原理

```
全量微调：W ∈ R^(d×k) 全部更新
LoRA低秩分解：不更新ΔW，而是学习两个小矩阵 A 和 B
  ΔW = B @ A，其中 B ∈ R^(d×r), A ∈ R^(r×k), r << min(d, k)
  W_new = W_old + B @ A
```

**参数量对比（d=4096, k=4096, r=8）**：

| 方式 | 参数量 | 压缩比 |
|------|--------|--------|
| 全量微调 | 16,777,216 | 1x |
| LoRA | 65,536 | **256倍** |

**为什么低秩有效**：

| 解释 | 说明 |
|------|------|
| **Intrinsic Rank** | 任务相关参数变化具有低秩结构 |
| **浪费理论** | 预训练模型已学大量知识，只需小调整 |
| **梯度SVD** | 对梯度做SVD，主要信息集中在少数奇异值上 |

#### LoRA vs 全量微调对比

| 维度 | 全量微调 | LoRA |
|------|----------|------|
| 可训练参数量 | 全部（如70B） | 少量（如8×2×d） |
| 显存占用 | 全部+梯度+优化器 | 只存A、B和优化器 |
| 训练速度 | 慢 | 快（2-3倍加速） |
| 效果 | 通常最好 | 接近全量（~95%） |
| 灾难遗忘 | 风险高 | 风险低（W冻结） |
| 多任务切换 | 需保存全套权重 | 只需换A、B |

### 41. DPO 训练中模型输出变长原因？如何控制生成长度？

**A：**

#### DPO（Direct Preference Optimization）

```
L = - log σ( log πθ(y_w|x) / πref(y_w|x) - log πθ(y_l|x) / πref(y_l|x) )
  y_w = preferred response（偏好回答）
  y_l = rejected response（拒绝回答）
```

#### 输出变长原因

| 原因 | 说明 |
|------|------|
| **负样本太短** | 负样本总是简单敷衍的回答，模型学会"越详细越好" |
| **奖励黑客** | 长回答更容易被偏好（长=详细=好？） |
| **数据偏差** | 偏好数据中正样本平均比负样本长 |

#### 解决方法

**1. 长度控制正则化**

```python
L = L_dpo + λ * |len(y) - target_len|²
```

**2. 平衡训练数据长度分布**

```
确保正负样本长度尽量接近，或在正负样本对中做长度归一化
```

**3. 推理时控制**

```python
# 方法1：max_tokens限制
output = model.generate(input, max_new_tokens=200)

# 方法2：length_penalty
output = model.generate(input, length_penalty=1.0, max_new_tokens=300)
# >1鼓励短，<1鼓励长

# 方法3：repetition_penalty
output = model.generate(input, repetition_penalty=1.2)
```

### 42. 大模型出现复读机现象的原因？训练问题还是推理问题？

**A：**

**表现**：
```
输入：介绍北京
输出：北京是中国的首都，北京是中国的首都，北京是中国的首都...
# 或
输出：故宫位于北京，...天安门位于北京，...天安门位于北京...
```

#### 原因分析

| 层面 | 原因 | 说明 |
|------|------|------|
| **训练** | 数据重复 | 预训练语料有大量重复文本 |
| **训练** | 标签平滑不足 | one-hot标签导致过度自信 |
| **推理** | Temperature=0 | 贪婪解码，总是选最高概率token |
| **推理** | Top-p过大 | 采样范围太大，容易回到重复轨道 |
| **推理** | 上下文诱导 | 之前的输出内容引导后续重复 |

#### 解决方法

**推理阶段**：

```python
generation_config = {
    "temperature": 0.7,           # 0.7-0.9，不要用0
    "top_p": 0.9,                # 不要设1.0
    "repetition_penalty": 1.2,   # 1.05-1.2
    "no_repeat_ngram_size": 3,   # 3-gram不重复
    "max_new_tokens": 512,       # 限制最大长度
}
```

**训练阶段**：

| 方法 | 说明 |
|------|------|
| 数据去重 | 预训练/微调时去除重复语料 |
| Unlikelihood Training | 对重复token施加负向损失 |
| 标签平滑 | KL散度损失替代交叉熵 |

#### 判断训练还是推理问题

| 判断方法 | 推理问题特征 | 训练问题特征 |
|----------|--------------|--------------|
| 换推理参数 | 调temperature/penalty有效 | 调整无效 |
| 换模型 | 不同模型表现不同 | 多个模型都有同样问题 |

### 43. SFT 微调后模型的通用能力出现遗忘时如何处理？

**A：**

**遗忘现象（Catastrophic Forgetting）**：
```
SFT前：能回答"水杯的作用"，能写代码，能做数学题
SFT后：能回答"产品经理的职责"，但不再会做数学题了
```

#### 解决思路

**1. 数据层面——混合训练**

```python
train_data = (
    sft_data * 1.0 +   # 任务数据
    alpaca_data * 0.3  # 通用能力数据（降权）
)
```

**2. 训练策略——LoRA/Prefix-tuning**

```
全量微调 → 灾难遗忘风险大
LoRA/Prefix-tuning → 风险小（冻结主权重）
```

**3. 正则化——保留预训练知识**

```python
# L2正则化
L = L_task + λ * ||W_new - W_old||²

# KL散度正则化
L = L_task + α * KL(π_new || π_old)
```

**4. 经验重放（Replay）**

```python
class MemoryReplayDataset:
    def __init__(self, sft_data, replay_ratio=0.2):
        self.replay_data = sample_from_pretraining(replay_ratio)
        self.sft_data = sft_data
```

| 场景 | 推荐方案 |
|------|----------|
| 资源有限，单一任务 | LoRA + 少量通用数据混合 |
| 需要强专业能力 | 全量微调 + KL正则化 |
| 需要保留多种能力 | 多任务联合训练 |
| 任务简单但数据少 | Prefix-tuning / Adapter |
