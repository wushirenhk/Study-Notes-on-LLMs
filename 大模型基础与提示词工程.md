## 1. AI基础认知

### 你对 AI 有哪些了解？

AI（人工智能）是让机器具有人类智能的技术。当前最火热的路线是**大语言模型（LLM）**，其核心特征：

| 特征 | 说明 |
|------|------|
| **涌现能力（Emergent Abilities）** | 模型规模突破临界点后，突然具备小模型没有的能力（如 Chain-of-Thought推理） |
| **上下文学习（In-Context Learning）** | 不需微调，直接在prompt中给出示例就能学会新任务 |
| **零样本/少样本（Zero/Few-Shot）** | 泛化到完全没见过的任务 |
| **Scaling Law** | 模型性能随参数、数据、计算量幂律提升 |

主流模型分类：
- **纯Decoder（Autoregressive）**：GPT系擅长生成，如GPT-4、Claude、LLaMA
- **Encoder-Decoder**：T5、BART擅长理解+生成，如Gemini
- **混合专家（MoE）**：GPT-4、Mixtral等，多专家路由省算力

### 如何解决 AI 数字计算不准确的问题

大模型的数值计算是公认弱点，原因在于：

```
LLM本质是"下一个token预测"，对数字的语义理解 ≠ 精确数学运算
"123.456 × 789.012" 在模型眼里可能只是两个字符串
```

**解决方案：**

1. **外部工具计算**
   - 最有效：让模型调用Python解释器或计算器执行精确运算
   - 示例：`你负责推理，调用tool计算结果，我只接收最终答案`

2. **提示词工程优化**
   - 链式思考（CoT）：`让我们一步步计算...`
   - 数字拆解：把多位数的乘法拆成多个一位数运算
   - 显式指令：`你的任务是推理，最终计算交给Python脚本`

3. **结构化输出**
   - 用JSON格式明确区分"推理过程"和"最终数值"
   - 让模型输出数学表达式而非直接输出数值结果

4. **模型微调**
   - 对特定数学任务（如金融计算）进行LoRA微调
   - 构造高质量数学推理数据集（如GSM8K、MATH）

5. **专用数学模型**
   - 使用数学专用模型如MathGPT、NuminaMath
   - 或让GPT-4调用Wolfram Alpha进行符号计算

**核心原则**：**让LLM做推理，控制权交给精确计算模块**

## 2. 提示词工程

### 说一下 Few-Shot 机制，解决了什么问题，原理是什么

**解决的问题：**
- 模型在未见过的任务上泛化能力差
- 无法通过少量样本学习新模式
- 避免微调的高成本

**原理：**

```
零样本（Zero-Shot）：
┌─────────────────────────────────────────┐
│ 用户: 把"今天天气真好"翻译成英文        │
│ LLM: Today's weather is really nice.    │
└─────────────────────────────────────────┘

少样本（Few-Shot）：
┌─────────────────────────────────────────┐
│ 用户: "我爱你" → I love you            │
│        "谢谢" → Thank you              │
│        "再见" → ___                    │  ← 模型从示例中推断模式
│ LLM: Goodbye.                          │
└─────────────────────────────────────────┘
```

**为什么有效：**
1. **任务隐式编码**：示例把"翻译任务"的本质编码进上下文
2. **分布拟合**：模型根据示例拟合输入-输出映射关系
3. **注意力激活**：示例激活了相关任务的内部表示

**最佳实践：**
- 示例数量通常2-5个，过多可能稀释主题
- 示例要多样化，覆盖边界case
- 格式一致，减少模型解析歧义

### 写提示词，开头和结尾怎么写，哪个权重高

**结构化模板：**

```markdown
【开头 - 任务定义】
# 角色：你是专业的金融分析师
# 任务：为投资决策提供分析

【中间 - 核心内容】
[用户问题/数据/背景]

【结尾 - 输出格式约束】
# 要求：
1. 先分析再给出结论
2. 用表格呈现数据对比
3. 保守估计风险

【可选 - 行为约束】
# 禁止：直接给投资建议
# 语气：专业、客观、有数据支撑
```

**权重分析：**

```
注意力分布（简化模型）：
┌────────────────────────────────────────────┐
│  开头的任务定义、角色设定     ████████  高  │  ← 早期token获得更多attention
│  中间的核心内容/示例          ██████    中  │
│  结尾的格式约束、行为指令     ████████  高  │  ← 末期token影响最终输出
└────────────────────────────────────────────┘

结论：开头和结尾权重都高，中间相对较低
```

**调换顺序的影响：**

| 调换方式 | 可能影响 |
|---------|---------|
| 开头↔结尾互换 | 任务定义被推到末尾，可能被"遗忘"；格式约束在开头可能导致过早收敛 |
| 分散式 | 示例放在开头和结尾，中间放问题，有助于强化模式学习 |

**经验法则：**
- **开头**：放最重要的身份/任务定义（高权重）
- **结尾**：放必须遵守的硬约束（高权重，因为接近生成位置）
- **中间**：放示例和背景信息

### 提示词有一万个字和十个字会有什么问题

**一万字提示词的问题：**

```
问题1：上下文溢出（Context Overflow）
- 超过模型上下文窗口上限（约128K-200K tokens）
- 即使没超限，过长上下文会稀释关键信息（松散注意力问题）

问题2：成本爆炸
- 输入token数直接影响API费用
- 1万字 ≈ 7000+ tokens，每次调用成本高

问题3：延迟增加
- 更长的上下文意味着更长的处理时间
- 用户体验下降（等待时间变长）

问题4：推理质量下降
- 书籍《Lost in the Middle》现象：模型对长上下文中
  中间部分的信息记忆最差
- 关键信息埋在长文本中间容易被忽略

问题5：角色/指令遗忘
- 开头定义的角色在长文本中可能被稀释
- 模型在生成过程中"忘记"了自己应该做什么
```

**十个字提示词的问题：**

```
问题1：歧义性高
- 模型需要猜测用户真实意图
- "计算价格" → 计算什么价格？哪个时间范围？

问题2：缺少约束
- 没有输出格式、语气、长度限制
- 模型可以自由发挥，结果不可控

问题3：泛化风险
- 模型可能按最常见的场景理解
- 但用户意图可能是长尾case
```

**最佳实践：**
- 提示词长度适中，**50-500字**通常效果最好
- 信息密度比长度重要：去除冗余，保留关键约束
- 超长文本考虑：压缩、拆分成多轮对话、使用RAG分段检索

### 一两百字的提示词复制两遍，有什么用

**效果分析：**

```
原始提示词（200字）：
"你是数据分析助手，帮我分析销售数据...
[约束1] [约束2] [约束3] ..."

复制两遍后（400字）：
"你是数据分析助手，帮我分析销售数据...
[约束1] [约束2] [约束3] ...
你是数据分析助手，帮我分析销售数据...
[约束1] [约束2] [约束3] ..."

实际作用：
┌──────────────────────────────────────────────────┐
│ ✅ 强化约束效果（类似Few-Shot的重复强调）         │
│ ✅ 在超长上下文中确保关键指令不被遗漏             │
│ ❌ 不增加新信息，浪费token                       │
│ ❌ 可能触发模型的"重复检测"机制导致输出截断       │
│ ⚠️ 某些模型对重复内容有惩罚（repetition penalty）│
└──────────────────────────────────────────────────┘
```

**更优替代方案：**

1. **Few-Shot示例**（复制示例而非重复指令）
   ```
   示例1: [销售数据A] → [分析结果A]
   示例2: [销售数据B] → [分析结果B]
   ```

2. **结构化强调**（用格式强化）
   ```
   【重要】请务必按以下格式输出...
   【警告】禁止输出XXX...
   ```

3. **分层指令**
   ```
   SYSTEM: 角色设定（持久生效）
   USER: 具体任务 + 一次性约束
   ```

## 3. Transformer与注意力机制

### 大模型基本算法知识：Transformer

**整体架构：**

```
                    Transformer
                        │
          ┌─────────────┴─────────────┐
          ▼                           ▼
       Encoder                     Decoder
    (理解型任务)                (生成型任务)
    BERT、T5                     GPT系列

典型Decoder-only架构（GPT系）：
┌─────────────────────────────────────────┐
│  Input: "今天天气很好"                   │
│         ↓                               │
│  Token Embedding + Positional Encoding  │
│         ↓                               │
│  ┌─ Layer × N ─────────────────────┐    │
│  │    Self-Attention (因果掩码)     │    │
│  │           ↓                     │    │
│  │    Feed-Forward Network         │    │
│  │           ↓                     │    │
│  │    Residual + LayerNorm          │    │
│  └──────────────────────────────────┘    │
│         ↓                               │
│  Output: 下一个token的概率分布            │
└─────────────────────────────────────────┘
```

**核心组件：**

| 组件 | 作用 | 关键参数 |
|------|------|---------|
| Self-Attention | 建模序列内token间依赖关系 | h(多头数), d_model |
| FFN | 非线性变换，存储知识 | dff（通常是4×d_model） |
| LayerNorm | 稳定训练 | ε |
| Positional Encoding | 注入序列位置信息 | 正弦/旋转编码 |

### Self-Attention 的流程

**数学本质**：Attention(Q,K,V) = softmax(QK^T / √d_k) × V

```
详细流程：

Step 1: 输入准备
┌─────────────────────────────────────────┐
│ 输入序列: [Token1, Token2, Token3]       │
│ 每个token → Query, Key, Value向量        │
│ Q = X · W_q                             │
│ K = X · W_k                             │
│ V = X · W_v                             │
└─────────────────────────────────────────┘
           ↓
Step 2: 计算注意力分数
┌─────────────────────────────────────────┐
│ scores = Q · K^T                        │
│ [t1→t1, t1→t2, t1→t3]                   │
│ [t2→t1, t2→t2, t2→t3]                   │
│ [t3→t1, t3→t2, t3→t3]                   │
└─────────────────────────────────────────┘
           ↓
Step 3: 缩放
┌─────────────────────────────────────────┐
│ scores = scores / √d_k                  │
│ (防止点积过大导致梯度消失)                │
└─────────────────────────────────────────┘
           ↓
Step 4: Softmax归一化
┌─────────────────────────────────────────┐
│ attention_weights = softmax(scores)     │
│ 每行和为1，表示对其他token的注意力权重    │
└─────────────────────────────────────────┘
           ↓
Step 5: 加权求和
┌─────────────────────────────────────────┐
│ output = attention_weights · V          │
│ 每个token的输出 = 加权求和所有token的V    │
└─────────────────────────────────────────┘
           ↓
Step 6: 多头注意力
┌─────────────────────────────────────────┐
│ 并行运行h个attention head               │
│ 每个head关注不同方面的关系               │
│ concat后通过W_o投影                      │
└─────────────────────────────────────────┘
```

**形象理解：**
- Query=当前token"在问什么"
- Key=所有token"在回答什么"
- Value=所有token的"实际内容"
- Attention = "我应该听取谁的建议"

### Encoder里的Self-Attention流程

**Encoder vs Decoder 的关键差异：**

```
Encoder Self-Attention（双向）：
┌─────────────────────────────────────────┐
│ 输入: "The cat sat on the mat"          │
│                                      [CLS]
│ Token:  The  cat  sat  on  the  mat     │
│          ↓   ↓   ↓   ↓    ↓    ↓       │
│ Attend to: ALL tokens (双向可见)        │
│                                       [CLS]可以 attending to 整个序列
└─────────────────────────────────────────┘

Decoder Self-Attention（因果/单向）：
┌─────────────────────────────────────────┐
│ 输入: "I love cats"                     │
│                                        ↑
│ Generation: I → love → cats → EOS      │
│ Token:  I   love  cats                 │
│          ↓   ↓    ↓                    │
│ Attend to: I | love | cats             │
│           自身 前序  前序+自身          │
│          (不能看未来)                   │
└─────────────────────────────────────────┘

Cross Attention（Decoder关注Encoder）：
┌─────────────────────────────────────────┐
│ Decoder Query ←→ Encoder Keys/Values    │
│                                        │
│ Decoder: 生成的token作为Query           │
│ Encoder: 源序列的K,V                    │
│ 实现: Decoder对源序列的"选择性注意"      │
└─────────────────────────────────────────┘
```

**Encoder结构（以BERT为例）：**
```
Input Embedding + Positional Encoding
        ↓
┌─ Layer × N ─────────────────────────┐
│  1. Multi-Head Self-Attention       │  ← 双向attention
│         ↓                           │
│  2. Add & LayerNorm                 │
│         ↓                           │
│  3. Feed-Forward                    │
│         ↓                           │
│  4. Add & LayerNorm                 │
└─────────────────────────────────────┘
        ↓
[CLS] Token Output → 用于分类任务
```

**Decoder结构（以GPT为例）：**
```
Input Embedding + Positional Encoding
        ↓
┌─ Layer × N ─────────────────────────┐
│  1. Masked Self-Attention           │  ← 因果掩码，不能看未来
│         ↓                           │
│  2. Add & LayerNorm                 │
│         ↓                           │
│  3. Cross Attention (to Encoder)    │  ← 可选，用于Encoder-Decoder模型
│         ↓                           │
│  4. Add & LayerNorm                 │
│         ↓                           │
│  5. Feed-Forward                    │
│         ↓                           │
│  6. Add & LayerNorm                 │
└─────────────────────────────────────┘
        ↓
Linear + Softmax → 下一个token概率
```

### KV Cache

**解决的问题：**
```
背景：自回归生成时，每个新token都需要重新计算全序列的attention
       这导致：生成10个token = 10 × 计算全序列 = O(N²) 冗余

KV Cache核心思想：缓存已计算的K、V，避免重复计算
```

**原理：**

```
无KV Cache的Attention计算（生成第3个token）：
┌─────────────────────────────────────────┐
│ tokens: [T1, T2, T3]                    │
│ 每个新token都要重新计算：                │
│ Q3·K1^T + Q3·K2^T + Q3·K3^T            │
│ 需要所有token的K、V                      │
└─────────────────────────────────────────┘

有KV Cache的Attention计算（生成第3个token）：
┌─────────────────────────────────────────┐
│ 之前已缓存: K1, V1, K2, V2              │
│ 当前计算:   Q3, K3, V3                  │
│ 新token只需: Q3·K1^T + Q3·K2^T + Q3·K3^T│
│ 缓存命中加速！                          │
└─────────────────────────────────────────┘

实际实现：
┌─────────────────────────────────────────┐
│ Cache={}                                │
│ for token in序列:                       │
│   Q = Wq @ x                           │
│   if cache中有K,V:                      │
│     K = concat([cache.K, k])           │  ← 增量追加
│     V = concat([cache.V, v])           │
│   else:                                 │
│     K, V = k, v                         │
│   cache.K, cache.V = K, V               │
│   output = attention(Q, K, V)           │
└─────────────────────────────────────────┘
```

**图示：**

```
            逐token生成
Token1 ──→ Token2 ──→ Token3 ──→ Token4 ──→ ...
  │          │          │          │
  ↓          ↓          ↓          ↓
计算K,V    复用K,V    复用K,V    复用K,V
          Cache↑    Cache↑    Cache↑
```

### Flash Attention

**解决的问题：**
```
标准Attention实现：O(N²) 显存占用
原因：S = QK^T 需要构建完整的 N×N 注意力矩阵

Flash Attention核心思想：
分块计算 + 在线softmax → O(N) 显存
```

**原理：**

```
标准实现 vs Flash Attention：

标准实现：
┌─────────────────────────────────────────┐
│ Q: [N × d], K: [N × d]                  │
│ S = QK^T: 完整 N×N 矩阵                  │
│ 显存: O(N²)                             │
└─────────────────────────────────────────┘

Flash Attention（分块）：
┌─────────────────────────────────────────┐
│ 把Q、K、V分成T块：                       │
│ Block1: Q1, K1, V1 → 部分attention     │
│ Block2: Q1, K2, V2 → 累加到结果        │
│ (在线softmax技巧归一化分块结果)          │
│                                         │
│ 显存: O(N) 只需要几块而非全矩阵          │
└─────────────────────────────────────────┘

Triton实现的关键：
for i in range(0, N, block_size):
    for j in range(0, N, block_size):
        for k in range(0, d, block_size):
            # 加载块到SRAM
            # 执行矩阵乘法
            # 写回HBM
```

**效果：**

| 指标 | 标准Attention | Flash Attention |
|------|--------------|-----------------|
| 显存复杂度 | O(N²) | O(N) |
| 时间复杂度 | O(N²) | O(N²) (I/O优化) |
| 速度 | 基准 | 2-4x加速 |
| 上下文长度 | 4K-32K | 16K-128K+ |

### Context有什么优化吗

**Context优化的主要方向：**

```
1. 位置编码优化
   ├── RoPE (Rotary Position Embedding) - LLaMA采用
   ├── ALiBi (Attention with Linear Biases) - BLOOM采用
   └── 原因：原始Sinusoidal/可学习PE在长上下文泛化差

2. 注意力机制优化
   ├── Flash Attention - 分块计算省显存
   ├── Ring Attention - 多GPU分布式注意力
   ├── Sparse Attention - 只关注重要位置
   └── Linear Attention - 线性复杂度近似

3. 上下文管理
   ├── KV Cache管理 - 缓存复用、淘汰策略
   ├── Context Compression - 压缩不重要token
   └── Chunking/Windowing - 分块处理长序列

4. 工程优化
   ├── Continuous Batching - 动态batch拼接
   ├── PagedAttention - KV Cache内存管理（vLLM）
   └── Tensor Parallelism - 张量并行切分
```

**vLLM的PagedAttention示例：**
```
传统KV Cache：连续显存分配，碎片化严重
PagedAttention：类OS分页管理，按需分配，显存利用率提升2-3x
```

## 4. RAG相关

### 什么是 RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

```
核心思想：让LLM在生成回答前，先从外部知识库检索相关信息

无RAG：                                    有RAG：
┌─────────────────┐                        ┌─────────────────┐
│ LLM             │                        │ Query           │
│ (依赖内部知识)   │                        │     ↓           │
│ ↓               │                        │ Retrieval       │
│ 生成回答        │                        │     ↓           │
│ (可能幻觉/过时) │                        │ 相关文档        │
└─────────────────┘                        │     ↓           │
                                           │ LLM + 上下文    │
                                           │     ↓           │
                                           │ 生成回答        │
                                           └─────────────────┘
```

**为什么需要RAG：**
- LLM知识有截止日期，无法回答实时问题
- LLM在垂直领域（企业内部知识）上可能空白
- 减少幻觉，提高回答准确性
- 可溯源，方便审核

### 既然知识时效性的问题可以用 RAG 解决，为什么不直接联网搜索呢？

**RAG vs 联网搜索 对比：**

| 维度 | RAG | 联网搜索 |
|------|-----|---------|
| **数据可控性** | 企业内部数据、隐私数据可用 | 依赖第三方搜索引擎 |
| **实时性** | 依赖索引更新频率，可近实时 | 通常最新 |
| **回答质量** | 可控的权威知识源 | 搜索结果质量参差 |
| **延迟** | 本地检索，低延迟 | 搜索+爬取，高延迟 |
| **成本** | 一次性建索引，后续免费 | 每次搜索都要付费 |
| **覆盖范围** | 只检索已有知识库 | 理论上无限网络 |
| **合规性** | 满足数据不出境的监管要求 | 可能涉及境外数据 |

**选择建议：**

```
场景1：企业客服、私域知识库
→ RAG首选（数据安全、定制化、低延迟）

场景2：实时新闻问答
→ 联网搜索首选（时效性强、覆盖面广）

场景3：既需要实时性又需要准确性
→ 混合架构：联网搜索 + RAG验证/补充
```

### RAG的整体流程是什么样？

```
完整RAG流程：

1. 文档处理（ indexing阶段）
┌─────────────────────────────────────────┐
│ PDF/Markdown/HTML/数据库                 │
│         ↓                               │
│ 文档解析（Layout解析、表格提取）          │
│         ↓                               │
│ 文本切分（Chunking）                     │
│   - 固定大小切分（token数）              │
│   - 语义切分（段落、句子）               │
│   - 层次切分（标题层级）                │
│         ↓                               │
│ 向量化（Embedding Model）                │
│         ↓                               │
│ 存储到向量数据库                         │
│ (同时保存原始文本到Metadata)             │
└─────────────────────────────────────────┘

2. 检索阶段（Retrieval）
┌─────────────────────────────────────────┐
│ 用户Query                                │
│         ↓                               │
│ Query向量化（Embedding）                 │
│         ↓                               │
│ 向量相似度搜索（Top-K）                   │
│   - 余弦相似度 / 内积 / L2距离           │
│         ↓                               │
│ 候选Chunk召回                           │
│         ↓                               │
│ 可选：Rerank（精排）                     │
│         ↓                               │
│ 可选：过滤/重排序（metadata条件）        │
└─────────────────────────────────────────┘

3. 生成阶段（Generation）
┌─────────────────────────────────────────┐
│ 检索到的相关文档                         │
│         ↓                               │
│ 构建Prompt                               │
│ [System: 你是一个助手，基于以下资料回答] │
│ [Context: 检索到的文档内容]              │
│ [Query: 用户问题]                        │
│         ↓                               │
│ LLM生成回答                             │
└─────────────────────────────────────────┘
```

### 在你的系统中RAG检索出来质量很低你觉得有哪些原因？以及该怎么解决？

**常见原因及解决方案：**

```
原因1：文档切分策略不当
┌─────────────────────────────────────────┐
│ 问题：chunk太小→上下文不完整            │
│      chunk太大→引人噪声，主题分散        │
│ 典型症状：检索到的片段缺少关键背景        │
│                                       │
│ 解决：                                   │
│ - 尝试多种chunk_size（256/512/1024）    │
│ - 语义切分而非固定大小                   │
│ - 保留重叠（如50% overlap）              │
│ - 层次切分：小chunk用于检索，大chunk用于 │
│   生成（Parent-Document Retrieval）     │
└─────────────────────────────────────────┘

原因2：Embedding模型不匹配
┌─────────────────────────────────────────┐
│ 问题：通用Embedding在垂直领域效果差      │
│ 典型症状：语义相关的内容被遗漏            │
│                                       │
│ 解决：                                   │
│ - 使用领域适配的Embedding模型            │
│ - 微调Embedding（少量标注数据）          │
│ - BGE-M3、text-embedding-3等最新模型     │
└─────────────────────────────────────────┘

原因3：Query和文档的语义不匹配
┌─────────────────────────────────────────┐
│ 问题：用户用词和文档用词存在差异          │
│ 例：用户搜"电脑" 文档写"计算机"         │
│                                       │
│ 解决：                                   │
│ - HyDE（假设性文档生成）                │
│ - Query改写（Rewrite-Retrieve-Read）     │
│ - 同义词扩展 / BM25 + 向量混合检索       │
└─────────────────────────────────────────┘

原因4：检索策略单一
┌─────────────────────────────────────────┐
│ 问题：只用向量检索，缺少语义层面的理解    │
│                                       │
│ 解决：                                   │
│ - 混合检索：向量 + BM25（关键词）        │
│ - Rerank模型二次排序                    │
│ - 多跳检索（Multi-Hop）                  │
│ - 元数据过滤 + 向量检索                 │
└─────────────────────────────────────────┘

原因5：向量数据库参数不当
┌─────────────────────────────────────────┐
│ 问题：HNSW的ef太低→召回不全              │
│      分区不合理→相关文档被分开           │
│                                       │
│ 解决：                                   │
│ - 调优HNSW的ef/ M参数                   │
│ - 选择合适的度量（余弦/内积）            │
│ - 注意归一化（余弦相似度必须）           │
└─────────────────────────────────────────┘
```

### 为什么要用 RAG

**RAG的核心价值：**

| 价值维度 | 说明 |
|---------|------|
| **知识时效性** | 索引更新即可，无需重训模型 |
| **可解释性** | 答案有来源可追溯 |
| **数据安全** | 私有数据不离库 |
| **成本可控** | vs 微调：无需重新训练LLM |
| **灵活性** | 可快速更新/切换知识源 |
| **减少幻觉** | 基于真实文档生成 |

**何时用RAG vs 微调：**

```
RAG适用场景：
✅ 需要频繁更新知识（新闻、产品变更）
✅ 数据源多样（多文档类型）
✅ 需要可解释性/溯源
✅ 知识库规模中等（<百万级）

微调适用场景：
✅ 任务/风格固定（客服话术、写作风格）
✅ 推理模式需要固化（如特定领域的Chain-of-Thought）
✅ 低延迟场景（减少prompt长度）
✅ 成本：大量相似query时分摊训练成本
```

### 如何评估 RAG 的效果

**评估指标体系：**

```
1. 检索阶段指标（Retrieval Metrics）

┌────────────────┬────────────────────────────────┐
│ 指标           │ 说明                            │
├────────────────┼────────────────────────────────┤
│ Hit@K          │ Top-K中是否有相关文档（1/0）    │
│ MRR            │ 相关文档首次出现位置的倒数平均   │
│ NDCG@K         │ 考虑位置的归一化折损累计增益     │
│ Recall@K       │ Top-K召回的相关文档比例         │
└────────────────┴────────────────────────────────┘

2. 生成阶段指标（Generation Metrics）

┌────────────────┬────────────────────────────────┐
│ 指标           │ 说明                            │
├────────────────┼────────────────────────────────┤
│ Faithfulness   │ 答案是否忠实于检索到的上下文     │
│ Answer Relevancy│ 答案和问题相关性如何            │
│ Context Precision│ 检索上下文中有多少是真正相关的 │
│ Context Recall │ 相关文档有多少被召回            │
└────────────────┴────────────────────────────────┘

3. 端到端评估（使用LLM评判）

RAGAS框架：
- Faithfulness: 答案事实是否从context推断
- Answer Relevancy: 答案对问题的帮助程度

评估流程示例：
┌─────────────────────────────────────────┐
│ 1. 准备标注数据集（query + ground truth）│
│ 2. 运行RAG系统得到 (context, response)   │
│ 3. 用GPT-4等强模型做LLM-as-Judge        │
│ 4. 计算指标分数                          │
└─────────────────────────────────────────┘

4. 实际业务指标

- 用户满意度 / 人工评估
- 答案采纳率
- 幻觉率（答非所问比例）
- 响应延迟
- 知识覆盖率（可回答的问题占比）
```

## 5. Agent与工具调用

### 对 MCP 有了解吗？什么是 MCP？

**MCP = Model Context Protocol（Anthropic提出的模型上下文协议）**

```
定位：让AI模型安全、受控地调用外部工具/数据的标准协议

类比：USB-C协议（设备连接标准）
      MCP = AI模型的"USB-C接口"
      各种工具/数据源 = USB设备
```

**核心架构：**

```
┌─────────────────────────────────────────────────┐
│                  Host Application               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────┐ │
│  │ Claude Code │  │ Claude App  │  │ 其他AI  │ │
│  └──────┬──────┘  └──────┬──────┘  └────┬────┘ │
│         │                │               │      │
│  ┌──────┴────────────────┴───────────────┴────┐ │
│  │           MCP Client (SDK)                  │ │
│  └──────┬────────────────┬───────────────┬────┘ │
└─────────┼────────────────┼───────────────┼──────┘
          │                │               │
┌─────────┼────────────────┼───────────────┼──────┐
│         │    MCP Protocol │               │      │
│  ┌──────┴──────┐  ┌──────┴──────┐  ┌─────┴─────┐ │
│  │ File System │  │ GitHub      │  │ Database  │ │
│  │   Server    │  │   Server    │  │  Server   │ │
│  └─────────────┘  └─────────────┘  └───────────┘ │
│        ↑                ↑               ↑        │
│   远程/本地文件    GitHub API      SQL查询       │
└─────────────────────────────────────────────────┘
```

**MCP核心概念：**

| 概念 | 说明 |
|------|------|
| **Host** | AI应用本身（如Claude Desktop） |
| **Client** | Host内的MCP客户端SDK |
| **Server** | 提供特定能力的轻量服务（文件/Git/数据库等） |
| **Protocol** | 客户端和服务器间的通信格式（JSON-RPC 2.0） |
| **Tools** | 服务器暴露的可调用函数 |
| **Resources** | 服务器提供的数据（文件内容、查询结果） |
| **Prompts** | 预定义的prompt模板 |

**与普通API调用的区别：**

```
普通工具调用（硬编码）：
┌─────────────────────────────────────────┐
│ LLM决定调用tool                         │
│ → 构造HTTP请求（API key、endpoint）     │
│ → 解析返回JSON                         │
│ 问题：每个工具都要单独集成，格式不统一   │
└─────────────────────────────────────────┘

MCP：
┌─────────────────────────────────────────┐
│ LLM决定调用tool                         │
│ → MCP Client标准化请求                  │
│ → MCP Server处理                       │
│ → 返回标准化结果                        │
│ 优点：一次集成，所有MCP Server通用      │
└─────────────────────────────────────────┘
```

### 在Agent里用MCP和用普通工具相比，有哪些不好的地方？

**MCP的潜在问题：**

```
问题1：额外抽象层 = 额外延迟
┌─────────────────────────────────────────┐
│ 纯工具调用: LLM → 直接HTTP → 工具响应   │
│ MCP调用:   LLM → MCP Client → 协议解析  │
│                   → MCP Server → 工具   │
│                   → 协议序列化 → 响应   │
│                   (多了2-3跳)           │
└─────────────────────────────────────────┘

问题2：增加上下文长度（见下详解）
┌─────────────────────────────────────────┐
│ MCP每次调用需要传递：                   │
│ - tool定义（名称、参数schema、描述）     │
│ - 工具执行结果                          │
│ - 可能多次往返（tool_use → result → ...）│
│ vs 普通tool：工具定义在系统prompt内复用  │
└─────────────────────────────────────────┘

问题3：状态管理复杂性
┌─────────────────────────────────────────┐
│ MCP Server可能有状态（session、连接池）  │
│ 普通工具通常无状态，天然可扩展           │
│ 需要处理：连接管理、错误重试、超时      │
└─────────────────────────────────────────┘

问题4：调试/追踪困难
┌─────────────────────────────────────────┐
│ 普通API调用：请求/响应明确可追踪         │
│ MCP：协议层隐藏细节，debug需要工具      │
└─────────────────────────────────────────┘

问题5：生态成熟度
┌─────────────────────────────────────────┐
│ 普通工具：SDK成熟（boto3、requests等）  │
│ MCP：还处于早期，Server实现参差不齐     │
└─────────────────────────────────────────┘
```

**何时选MCP vs 普通工具：**

```
选MCP：
✅ 需要标准化集成多个工具（文件+Git+数据库+...）
✅ 工具需要状态（数据库连接、Git操作）
✅ 快速原型 / Claude Desktop场景

选普通工具：
✅ 对延迟敏感（生产级高性能需求）
✅ 工具数量少，集成简单
✅ 已有成熟的SDK
```

### 为什么MCP会导致上下文长度特别长？

**核心原因：MCP的"描述即代码"模式**

```
普通API调用：
┌─────────────────────────────────────────┐
│ 系统prompt（一次性）：                    │
│ "你有这些工具可用：[tool1, tool2, tool3]"│
│                                           │
│ 运行时（每轮对话）：                      │
│ user: query                              │
│ assistant: <tool_call>                   │
│ tool: result                             │
│ assistant: response                      │
│                                           │
│ 上下文增长：O(tool_calls)                │
└─────────────────────────────────────────┘

MCP：
┌─────────────────────────────────────────┐
│ 每次tool_use（Anthropic API格式）：       │
│ {                                       │
│   "role": "assistant",                  │
│   "content": [{                        │
│     "type": "tool_use",                │
│     "name": "filesystem_read_file",    │
│     "input": {"path": "xxx"},          │
│     // 完整的参数定义！可能很长         │
│   }]                                    │
│ }                                       │
│                                           │
│ 每次tool_result：                       │
│ {                                       │
│   "role": "user",                      │
│   "content": [{                        │
│     "type": "tool_result",            │
│     "tool_use_id": "xxx",             │
│     "content": "文件内容..."          │
│     // 实际执行结果，可能很大           │
│   }]                                    │
│ }                                       │
│                                           │
│ 上下文增长：O(tool_calls × (定义+结果))  │
│ 比普通tool大5-10x                       │
└─────────────────────────────────────────┘
```

**具体计算示例：**

```
假设一个MCP工具调用：

普通API格式（约200 tokens）：
{
  "type": "function",
  "function": {
    "name": "read_file",
    "arguments": {"path": "/a/b/c.txt"}
  }
}

MCP格式（可能1000+ tokens）：
{
  "type": "tool_use",
  "id": "toolu_01J9X...",
  "name": "read_file",
  "input": {
    "path": {
      "type": "string",
      "description": "Path of the file to read",
      "required": true
    },
    "offset": {
      "type": "number",
      "description": "Line offset to start reading from",
      "default": 0
    },
    "limit": {
      "type": "number",
      "description": "Maximum number of lines to read",
      "default": 10000
    }
  }
}

加上执行结果后差距更大
```

### MCP和上下文历史之间有什么关系？

**关系分析：**

```
MCP Tool Call在上下文中的生命周期：

1. Tool Definition阶段（一次性）：
   - MCP Server连接时，tools列表被加入system prompt
   - 每个tool包含完整JSON Schema（可能是几KB）
   - 仅在连接时计入上下文一次

2. Tool Use阶段（每次调用）：
   ┌─────────────────────────────────────────┐
   │ assistant消息：                        │
   │ - tool_use块包含：完整参数结构（重）   │
   │ - 如果MCP工具参数复杂，一个use可能1-2KB │
   │                                         │
   │ user消息（tool_result）：              │
   │ - 实际执行结果                        │
   │ - 可能是文件内容（很大！如果是读文件）│
   │                                         │
   │ 关键问题：这个结果被保留在对话历史中    │
   │           直到会话结束或被摘要         │
   └─────────────────────────────────────────┘

3. 对话历史的累积效应：
   ┌─────────────────────────────────────────┐
   │ 对话轮次 | 普通Tool累计 | MCP累计       │
   │    5    |   ~2KB       |   ~15KB       │
   │   20    |   ~8KB       |   ~60KB       │
   │   50    |   ~20KB      |   ~150KB     │
   │  100    |   ~40KB      |   ~300KB+    │
   │                                         │
   │ (粗略估算，MCP上下文增长是普通tool的5-8x)│
   └─────────────────────────────────────────┘
```

**缓解策略：**

```
策略1：结果截断/摘要
- 工具返回大结果时，截断或LLM摘要后保留
- 保留关键信息，去除冗余细节

策略2：消息压缩
- 定期对对话历史做摘要（如果模型支持）
- 把早期tool交互压缩为结论

策略3：工具设计优化
- 避免返回原始文件内容，改返回结构化摘要
- 多次小文件读取 > 一次大文件读取

策略4：上下文窗口管理
- 定期清理/截断历史
- 使用滑动窗口保留最近N轮

策略5：改用普通API（延迟敏感场景）
- 减少协议开销和上下文膨胀
```

### 你搭过Agent吗？做过Agent相关项目吗？

**Agent核心架构：**

```
┌─────────────────────────────────────────────────┐
│                    Agent架构                    │

│  ┌─────────────┐                                │

│         ↓                                       │
│  ┌─────────────┐                                │
│  │   Tools    │  ← 工具选择与调用               │
│  └──────┬──────┘                                │
│         ↓                                       │
│  ┌─────────────┐                                │
│  │  Memory    │  ← 记忆管理（短期+长期）         │
│  └──────┬──────┘                                │
│         ↓                                       │
│  ┌─────────────┐                                │
│  │  Execution  │  ← 行动执行与结果处理           │
│  └──────┬──────┘                                │
│         ↓                                       │
│  ┌─────────────┐                                │
│  │  Evaluation │  ← 效果评估，决定是否重试/回退  │
│  └─────────────┘                                │
└─────────────────────────────────────────────────┘
```

**主流Agent范式：**

```
1. ReAct (Reasoning + Acting)
┌─────────────────────────────────────────┐
│ Thought → Action → Observation → ...   │
│ 思考      执行     观察      循环       │
│                                       │
│ 适合：需要推理+工具调用的复杂任务       │
└─────────────────────────────────────────┘

2. Plan-and-Execute
┌─────────────────────────────────────────┐
│ Planner: 生成完整计划                   │
│ Executor: 按顺序执行                    │
│ 适合：多步骤、需全局规划的任务         │
└─────────────────────────────────────────┘

3. AutoGPT/ReAct风格
┌─────────────────────────────────────────┐
│ 自主循环：思考→行动→反思→决定下一步     │
│ 适合：开放性探索任务                   │
│ 风险：可能陷入循环/浪费token           │
└─────────────────────────────────────────┘
```

### 龙虾（OpenClaw）你装过吗？它的记忆机制能讲一下吗？分几层记忆，每层是怎么实现的？

**OpenClaw记忆机制（分層记忆架构）：**

```
┌─────────────────────────────────────────────────┐
│            OpenClaw 记忆架构                    │
│                                                 │
│  Layer 1: Working Memory（工作记忆）            │
│  ├─ 位置：LLM的上下文窗口                       │
│  ├─ 内容：当前session的对话历史                 │
│  ├─ 容量：受模型上下文限制（如128K tokens）     │
│  └─ 淘汰：session结束后清除                     │
│                                                 │
│  Layer 2: Episodic Memory（情景记忆）           │
│  ├─ 位置：向量数据库（如Chroma/Pinecone）       │
│  ├─ 内容：历史会话的摘要/重要片段               │
│  ├─ 检索：基于语义相似度召回                   │
│  └─ 淘汰：长期不使用可能被遗忘                  │
│                                                 │
│  Layer 3: Semantic Memory（语义记忆）          │
│  ├─ 位置：向量数据库                           │
│  ├─ 内容：提取的事实、知识、偏好               │
│  ├─ 来源：从episodes中提取、结构化存储         │
│  └─ 特点：与具体事件解耦的抽象知识              │
│                                                 │
│  Layer 4: Procedural Memory（程序记忆）        │
│  ├─ 位置：配置/知识库文件                       │
│  ├─ 内容：工具使用模式、工作流、SOP            │
│  ├─ 更新：需要时显式更新配置                    │
│  └─ 持久：最稳定的记忆层                       │
└─────────────────────────────────────────────────┘

检索流程（记忆召回）：
┌─────────────────────────────────────────────────┐
│ 用户Query                                       │
│     ↓                                          │
│ 1. 查Working Memory（当前上下文）               │
│     ↓ 未命中                                   │
│ 2. 查Episodic Memory（向量检索最近对话）        │
│     ↓ 未命中                                   │
│ 3. 查Semantic Memory（提取的知识点）           │
│     ↓ 未命中                                   │
│ 4. 查Procedural Memory（工具/工作流）           │
│     ↓ 全部未命中                               │
│ 5. 纯LLM推理/请求用户澄清                      │
└─────────────────────────────────────────────────┘

记忆写入流程：
┌─────────────────────────────────────────────────┐
│ 当前Session结束                                 │
│     ↓                                          │
│ 1. 摘要生成：LLM生成会话摘要                   │
│     ↓                                          │
│ 2. 知识提取：从中提取语义知识到Semantic层       │
│     ↓                                          │
│ 3. 向量化存储：存入 Episodic/Semantic DB        │
└─────────────────────────────────────────────────┘
```

## 6. 向量与检索

### 向量是怎么检索的，大概结构是什么样的

**向量检索核心思想：**

```
把文本/图像等内容转为向量，在向量空间中找"距离最近"的邻居

文本 → 向量（通常768-1536维）→ 向量数据库
                                    ↓
用户Query → 向量 → 相似度搜索 → Top-K最近邻
```

**常用相似度度量：**

```
┌─────────────┬────────────────────────────────┐
│  余弦相似度  │ cos(θ) = A·B / (|A||B|)       │
│              │ 取值[-1, 1]，方向相似为1        │
├─────────────┼────────────────────────────────┤
│  内积 (IP)   │ A·B = ΣAi×Bi                   │
│              │ 未归一化时越大表示越相似        │
├─────────────┼────────────────────────────────┤
│  L2欧氏距离  │ ||A-B|| = √(Σ(Ai-Bi)²)         │
│              │ 越小表示越接近                  │
└─────────────┴────────────────────────────────┘

注意：向量检索前必须做L2归一化，否则余弦相似度会失效
```

**向量数据库的索引结构（近似最近邻 - ANN）：**

```
1. 暴力搜索（Brute Force）
┌─────────────────────────────────────────┐
│ 对所有向量计算距离，排序取Top-K          │
│ 时间：O(N)  N=向量数量                   │
│ 缺点：百万级数据极慢                     │
└─────────────────────────────────────────┘

2. HNSW（Hierarchical Navigable Small World）
┌─────────────────────────────────────────┐
│ 分层图结构：                            │
│ Level 3: 稀疏高速导航层（长距离跳跃）   │
│ Level 2:                                    │
│ Level 1:                                    │
│ Level 0: 密集底层（精确近邻）           │
│                                        │
│ 搜索：从顶层入口，贪婪遍历到目标附近      │
│ 复杂度：O(log N)                       │
│ 参数：M（每层连接数），ef（搜索宽度）   │
└─────────────────────────────────────────┘

3. IVF（Inverted File Index）
┌─────────────────────────────────────────┐
│ 聚类 + 倒排索引：                        │
│ 1. 用K-Means把向量分成N个簇（Centroid） │
│ 2. 建立倒排索引：簇ID → 簇内向量列表     │
│ 3. 搜索：定位最近的K个簇，遍历簇内向量   │
│ 缺点：需要额外参数调优（nprobe）         │
└─────────────────────────────────────────┘

4. PQ（Product Quantization）
┌─────────────────────────────────────────┐
│ 把高维向量分段压缩：                     │
│ 768维 → 分成8段，每段96维               │
│ 每段独立K-Means聚类（典型256类）        │
│ 存储：用各段的聚类ID替代原始向量         │
│ 压缩比：768×4 bytes → 8×1 byte = 8 bytes│
│ 搜索：解压缩后比较（或用优化算法）       │
└─────────────────────────────────────────┘

实际系统通常组合使用：
- HNSW：快速ANN搜索
- IVF：聚类过滤
- PQ：压缩存储（FAISS常用）
```

### 假如有一百万个向量数据集，怎么构建索引，检索流程是什么样的

**大规模向量检索架构：**

```
┌─────────────────────────────────────────────────────────────┐
│                 100万向量检索系统                           │

│  │    100万文档 → 100万 × 768 × 4 bytes = 3GB        │   │
│  │                                                      │   │
│  │ 2. 归一化（L2 Normalization）                       │   │
│  │    确保余弦相似度可用                               │   │
│  │                                                      │   │
│  │ 3. 构建HNSW索引                                    │   │
│  │    M=16（每层16条边）                              │   │
│  │    efConstruction=200（建索引宽度）                │   │

│  └─────────────────────────────────────────────────────┘   │

│  │                                                      │   │

│  │  - 分片（Shard）：向量按ID哈希分区                   │   │
│  │  - 副本（Replica）：多副本高可用                     │   │

│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  阶段3: 查询流程                                           │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  用户Query                                           │   │
│  │      ↓                                               │   │
│  │  1. Query向量化（Embedding）                       │   │
│  │      耗时：~50ms（GPU快，CPU慢）                    │   │
│  │      ↓                                               │   │
│  │  2. 路由层（Route）                                  │   │
│  │      - 随机路由（简单负载均衡）                      │   │
│  │      - 流量控制（防止过载）                          │   │
│  │      ↓                                               │   │
│  │  3. 并行搜索各分片                                   │   │

│  │      Shard2: HNSW.search(query, k=100) → 100个结果│   │
│  │      Shard3: ...                                     │   │
│  │      ↓合并（Merge Sort by score）                    │   │
│  │  4. Rerank（可选）                                   │   │
│  │      - 用Cross-Encoder精排（质量更高但慢）           │   │
│  │      或                                             │   │
│  │      - 直接返回Top-K                                │   │
│  │      ↓                                               │   │
│  │  5. 返回结果                                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  性能指标（参考）：                                        │
│  ┌──────────────────┬────────────┬────────────┐           │
│  │ 数据规模          │ 延迟(P99)  │ QPS        │           │
│  ├──────────────────┼────────────┼────────────┤           │
│  │ 100万向量        │ <50ms     │ ~1000      │           │
│  │ 1000万向量       │ <100ms    │ ~500       │           │
│  │ 1亿向量（分布式）│ <200ms    │ ~200       │           │
│  └──────────────────┴────────────┴────────────┘           │
└─────────────────────────────────────────────────────────────┘
```

### 向量化是怎么做的

**Embedding流程：**

```
文本 → Tokenize → Model Forward → Dense Vector

具体步骤：
┌─────────────────────────────────────────────────┐
│ 1. Tokenize                                     │
│    "今天天气很好" → [今天, 天气, 很, 好]         │
│    → [101, 2023, 4025, 3456, 102] (BERT中文)    │
│                                                 │
│ 2. Model Forward（Transformer Encoder）        │
│    tokens → Embedding层 → Self-Attention × N   │
│                                                 │
│ 3. 提取向量表示                                 │
│    - [CLS] token向量（句向量）                  │
│    - Mean Pooling所有token向量                  │
│    - Attention加权pooling                       │
│                                                 │
│ 4. 归一化                                       │
│    v = v / ||v||                               │
└─────────────────────────────────────────────────┘
```

**主流Embedding模型：**

| 模型 | 维度 | 特点 | 适用场景 |
|------|------|------|---------|

| text-embedding-3-large | 3072 | OpenAI最新 | 通用 |
| Jina-v3 | 1024 | 中英文强 | 中英文混合 |
| M3E | 768 | 开源，中文优化 | 中文场景 |
| GTE | 768 | 阿里，开源 | 中文 |
| Voyager | 2560 | 学术benchmark领先 | 英文学术 |

```
│  │                                                      │   │
│  │  B. 预计算+缓存                                      │   │
│  │     - 热门路线提前算好                              │   │

│  │                                                      │   │
│  │  C. 异步回调模式                                     │   │
│  │     司机请求 → 队列 → 稍后通知结果                  │   │
│  │     不是同步等待，是异步通知                        │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**具体建议：**

```
核心观点：不是所有计算都需要LLM实时推理

分层策略：
┌─────────────────────────────────────────┐
│ LLM处理（少量）：                       │
│ - 异常case、司机申诉                   │
│ - 复杂组合定价（多因素综合）           │
│                                         │
│ 规则引擎处理（大量/常规）：            │
│ - 基础距离×单价                        │
│ - 时段系数、天气系数                   │
│ - 历史数据统计定价                     │
└─────────────────────────────────────────┘

对于真正需要LLM的场景：
- 批处理代替实时
- 用小模型（如Qwen-7B）代替大模型
- 优化prompt，减少token
```

### AI操作DB有两个流派，一个是让AI生成sql，一个是让AI生成一段数据分析脚本，执行脚本，哪个好

**两派对比：**

```
┌──────────────────┬──────────────────────┬──────────────────────┐
│ 维度             │ SQL生成派             │ 脚本执行派           │
├──────────────────┼──────────────────────┼──────────────────────┤
│ 原理             │ LLM生成SQL            │ LLM生成Python/JS     │
│                  │ → 执行SQL → 返回结果  │ → 执行脚本 → 返回结果│
├──────────────────┼──────────────────────┼──────────────────────┤
│ 表达能力         │ 受SQL语法限制          │ 几乎图灵完备         │
│                  │ 复杂分析需要嵌套查询   │ 任意数据操作         │
├──────────────────┼──────────────────────┼──────────────────────┤
│ 安全性           │ 需严格SQL注入防护      │ 更危险（代码执行）   │
│                  │ 建议：只读账户+权限    │ 必须沙箱化           │
├──────────────────┼──────────────────────┼──────────────────────┤
│ 准确性           │ SQL执行结果确定        │ 脚本执行可能出错     │
│                  │ 但LLM生成SQL可能语法错 │ 需要try-catch       │
├──────────────────┼──────────────────────┼──────────────────────┤
│ 可审计性         │ SQL语句可审查          │ 脚本逻辑需仔细审查   │
│                  │ 权限控制成熟           │ 数据泄露风险更大     │
├──────────────────┼──────────────────────┼──────────────────────┤
│ 适用场景         │ 固定报表、统计查询     │ 复杂分析、机器学习   │
│                  │ OLAP场景              │ 数据清洗、可视化     │
└──────────────────┴──────────────────────┴──────────────────────┘
```

**实践建议：**

```
混合策略（最优解）：

┌─────────────────────────────────────────────────────┐
│ 简单查询：SQL生成派                                  │
│ "今天有多少笔订单？"                                 │
│ → SELECT COUNT(*) FROM orders WHERE ...             │
│                                                     │
│ 复杂分析：脚本执行派                                 │
│ "分析本月订单趋势并预测下月"                         │
│ → 生成Python脚本（含统计/ML预测）                   │
│                                                     │
│ 安全加固：                                          │
│ 1. SQL派：read-only角色 + 查询超时 + 结果行数限制  │
│ 2. 脚本派：沙箱执行（Docker/Function）              │
│           + 禁用网络/文件写入                       │
└─────────────────────────────────────────────────────┘
```

### 真实场景，老板需要考勤打卡数据、报销数据，怎么算

**实际业务计算场景分析：**

```
需求拆解：
┌─────────────────────────────────────────────────────┐
│ 考勤数据：                                          │
│ - 打卡记录（时间、地点、设备）                      │
│ - 排班表（正常班、倒班、请假）                      │
│ - 计算：出勤天数、迟到次数、请假天数               │
│                                                     │
│ 报销数据：                                         │
│ - 发票明细（金额、类型、日期）                     │
│ - 报销标准（差旅、餐补、交通）                     │
│ - 计算：应报销、已报销、待审批、驳回               │
└─────────────────────────────────────────────────────┘

计算方式：规则引擎 > AI
┌─────────────────────────────────────────────────────┐
│ AI适合：                                            │
│ - 理解模糊需求："我的报销大概什么时候能下来"        │
│ - 解释异常："为什么我这个月报销这么少"            │
│ - 非结构化数据：拍照上传发票→提取信息              │
│                                                     │
│ 规则引擎适合：                                      │
│ - "迟到30分钟内扣50元"（精确规则）                 │
│ - 固定公式："应发工资 = 基本工资 + 绩效 - 扣款"   │

└─────────────────────────────────────────────────────┘

数据流：
┌─────────────────────────────────────────────────────┐
│ 打卡机/审批系统 → 数据仓库 → 规则引擎计算 → BI    │
│                                    ↓               │
│                            AI助手（查询/解释）     │
└─────────────────────────────────────────────────────┘
```

### 给你一个场景，你的系统突然收到大量的请求要同时处理，你觉得会遇到哪些瓶颈和采取哪些措施？

**瓶颈分析：**

```
┌─────────────────────────────────────────────────────────────┐
│                  突发流量 → 瓶颈链路                        │
│                                                             │
│  瓶颈1：API Gateway层                                       │
│  ├─ 问题：连接数上限、限流算法瓶颈                         │
│  ├─ 表现：拒绝连接、504 Timeout                            │
│  └─ 措施：水平扩展Gateway、限流降级                         │
│                                                             │
│  瓶颈2：LLM Inference                                        │
│  ├─ 问题：GPU算力固定，请求排队                             │
│  ├─ 表现：P99延迟飙升、queue积压                            │
│  └─ 措施：请求排队、Batching合并、自动扩缩容                │
│                                                             │
│  瓶颈3：向量数据库                                           │
│  ├─ 问题：并发检索性能下降                                 │
│  ├─ 表现：检索超时                                         │
│  └─ 措施：连接池、读写分离、读写分离+分片                  │
│                                                             │
│  瓶颈4：后端传统DB（MySQL/PostgreSQL）                     │
│  ├─ 问题：连接池耗尽、读写热点                             │
│  ├─ 表现：SQL超时、连接拒绝                                 │
│  └─ 措施：连接池调优、读写分离、缓存穿透优化               │
│                                                             │
│  瓶颈5：网络带宽                                             │
│  ├─ 问题：跨机房/跨区域流量瓶颈                            │
│  ├─ 表现：网络延迟增加                                      │
│  └─ 措施：CDN、就近接入、数据预加载                        │
│                                                             │
│  瓶颈6：内存/显存                                            │
│  ├─ 问题：OOM、显存不足                                    │
│  ├─ 表现：服务重启、CUDA OOM                                │
│  └─ 措施：KV Cache管理、Chunk压缩、OOM保护                 │
└─────────────────────────────────────────────────────────────┘
```

**应对措施体系：**

```
第一层：预防（容量规划）
┌─────────────────────────────────────────┐
│ - 提前压测，知道系统瓶颈点               │
│ - 预留30-50%冗余容量                     │
│ - 监控关键指标（提前告警）               │
└─────────────────────────────────────────┘

第二层：拦截（限流熔断）
┌─────────────────────────────────────────┐
│ - API Gateway限流（单机QPS限制）         │
│ - 服务降级（非核心功能熔断）             │
│ - 排队机制（带超时和优先级）             │
└─────────────────────────────────────────┘

第三层：弹性（自动扩缩）
┌─────────────────────────────────────────┐
│ - K8s HPA（基于CPU/自定义指标）          │
│ - vLLM自动扩缩                          │
│ - Kafka Consumer动态扩缩                │
└─────────────────────────────────────────┘

第四层：疏导（流量控制）
┌─────────────────────────────────────────┐
│ - 请求合并（Similar query batching）    │
│ - 结果缓存（相同请求直接返回）           │
│ - 异步处理（不实时返回的走队列）         │
└─────────────────────────────────────────┘

第五层：恢复（故障恢复）
┌─────────────────────────────────────────┐
│ - 幂等设计（重试不重复处理）             │
│ - 健康检查 + 自动摘除故障节点            │
│ - 限流后的用户提示（排队进度）           │
└─────────────────────────────────────────┘
```
